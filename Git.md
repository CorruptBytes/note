# 概述

Git是一个开源的分布式版本控制系统。

## 核心概念

<h3>仓库(<code>Repository</code>)</h3>

仓库可以理解为被`Git`管理的目录，其中的所有文件都可以被`Git`管理并跟踪其变化，以便追踪历史或还原到之前的某个版本。

仓库创建后，会在主目录生成一个`.git`隐藏文件夹，它保存了`Git`仓库的所有信息，如果删除这个文件夹，仓库也就被删除了

- 如果文件夹为空，`Git`不会将这个文件夹纳入版本控制

<h2>工作区(<code>Working Directory</code>)</h2>

又称为工作目录或本地工作目录，就是本地电脑上仓库的主目录(`.git`所在的目录)

<h2>暂存区(<code>Staging Area/Index</code>)</h2>

是一种临时存储区域，用于保存即将提交到Git仓库的修改内容。位于`.git/index`

<h2>本地仓库(<code>Local Repository</code>)</h2>

位于本地的仓库。

当提交修改时，需要先通过`git add`将提交到暂存区，再通过`git commit`提交到本地仓库。

<h2><code>.gitignore</code></h2>

仓库中的一个特殊文件，用于忽略一些不应该被加入仓库中管理的文件。

通常以下文件需要被忽略：

- 日志文件和文件夹
- 所有.class文件
- 所有.o文件
- 所有.env文件
- 所有.zip和tar文件
- 所有.pem文件

`.gitingore`中直接声明需要被忽略的文件名，一行一个，可以使用通配符，也可以使用`文件夹/`忽略整个文件夹。

空行或者以#开头的行会被Git忽略。一般空行用于可读性的分隔，#一般用作注释使用标准的Blob模式匹配，例如：星号＊通配任意个字符问号？匹配单个字符中括号[]表示匹配列表中的单个字符，比如：[abc]表示a/b/c

已经被添加到仓库管理的文件不会受`.gitignore`的影响。

<h3>远程仓库</h3>

Git中的远程仓库和本地仓库是完全独立的，双方的修改均不会影响对方。因此`Git`中提供了两个命令用于远程仓库和本地仓库进行同步:`git push`，`git pull`。

<h2>文件状态</h2>

Git将仓库中的文件分为四种状态：

- 未跟踪(Untrack):没有被纳入版本管理的文件，新建的文件默认不被纳入版本管理，需要添加缓存区后再提交到仓库。
- 未修改(Unmodified):已经被Git管理的但是文件内容还没有发生变化的文件
- 已修改(Modified):被Git管理，已被修改但是未将修改添加到暂存区的文件
- 已暂存(Staged):被Git管理，已被修改并且修改已经添加到暂存区的文件。

<h3>分支(<code>branch</code>)</h3>

分支是一个仓库的不同版本，可以独立存在，并且有自己的提交记录。

# 命令

Git中的所有命令都以`git`开头，后跟具体的操作,如`git init`为初始化一个仓库。

<h2>配置(<code>git config</code>)</h2>

```
git config [--global|--system] 配置键名 "配置内容"
```

- 修改配置
- `--global`表示修改全局配置，对所有仓库生效，`--system`表示修改系统配置，对所有用户生效;省略则表示本地配置，只对当前仓库生效
- 如果配置内容中不存在特殊字符(如空格),双引号可以省略

<h4>配置用户名与邮箱</h4>

```bash
git config --global user.name "userName"
git config --global user.email "address"
```

- 配置用户名和邮箱后，才能直到是谁提交了更改

<h4>保存用户名与密码</h4>

```bash
git config --global credential.helper store
```

<h4>查看配置信息</h4>

```bash
git config [--global|--system] --list
```

<h2>查看仓库状态</h2>

```
git status
```

- 可查看信息包括当前处于仓库的哪个分支，仓库中有哪些文件以及这些文件处于什么状态,以及当前暂存区中存在哪些变化。

<h2>将文件添加到暂存区</h2>

```
git add <fileName>
```

- `add`命令支持通配符，同时添加多个文件，支持`*`(匹配多个字符)
- 还可以接收一个文件夹，会提交文件夹中的所有文件
- 不能添加工作区之外的文件，会报错。

```
git rm --cached <file>
```

<h2>将暂存区的修改提交到仓库</h2>

```
git commited  [-m 提交信息]
```

- 如果不指定`-m`参数，会进入一个交互式界面手动输入提交信息(默认是vim)
- 提交信息不能为空，否则提交失败
- 每次提交都会生成一个唯一的提交ID，为一个字符串。

<h2>查看提交日志</h2>

```
git log [--oneline]
```

- 可以查看每次提交的ID，用户，提交日期和提交信息

<h2>回退仓库状态为之前某一次提交的状态</h2>

```
git reset [--soft | --hard | --mixed] [commitId]
```

- `--mixed`：默认参数,回退版本时，保留工作区的修改内容，丢弃暂存区的修改内容
- `--soft`:会将 HEAD 指针回退到指定提交，但保留暂存区和工作区。用于合并多次提交
  - 当提交多次小版本时，想要合并为一个大版本，可以回退到第一次小版本提交之前，再次提交。
  - 被撤销提交中的修改也保留在暂存区中，就相当于`--mixed`回退之后，通过`git add`添加了所有修改到暂存区的状态。
- `--hard`:回退版本时，丢弃工作区的修改内容和暂存区的修改内容
- 可以使用`HEAD^`表示上一个版本

<h2>查看文件差异</h2>

```
git diff [HEAD] <commitID1> <commitID2> [fileName]
```

- 可以查看文件在工作区，暂存区和本地仓库之间的差异。也可以查看文件在两个版本之间的差异；也可以查看文件在不同分支间的差异。
- 默认查看工作区和暂存区之间的差异,`HEAD`参数查看工作区和仓库之间的差异。
- 可以使用`HEAD`表示当前版本，`HEAD^`或`HEAD~`表示上一个版本,`HEAD~N`表示上n个版本

从工作区和暂存区删除文件

```
git rm <fileName>
```

<h2>远程仓库操作</h2>

**将本地仓库的修改推送到远程仓库并关联远程分支和本地分支**

```
git push -u <远程仓库名> <远程分支名>:<本地分支名>
```

- `git push -u origin main` 在推送代码的同时，会为当前本地分支设置上游分支为 `origin/main`。设置完成后，Git 会在配置中记录该映射关系，使得后续可以直接使用 `git push` 和 `git pull`，无需再显式指定远程仓库和分支名。

**将远程仓库的内容拉取到本地仓库**

```
git pull <远程仓库名> <远程分支名>:<本地分支名>
```

- `<远程分支名>:<本地分支名>`如果分支名相同，可以简写为`分支名`。
- 当 pull 的远程分支名与当前分支名一致，且当前分支未设置 upstream 时，Git 会将其设为 upstream。

**关联本地仓库与远程仓库**

1. 首先需要先创建一个远程仓库

2. 执行命令关联仓库

   ```
   git remote add origin url
   ```

- `origin`为远程仓库默认的别名
- 默认`clone`而来的仓库与远程仓库本身就具有关联，别名为`origin`

3. 关联本地分支与远程分支

   ```
   git pull <远程仓库名> <远程分支名>:<本地分支名>
   ```

   - `git clone`时会默认将本地当前分支与远程默认分支关联

**查看与当前仓库关联的远程仓库信息**

```
git remote -v
```

- 输出内容为多行行

  ```
  <远程仓库名> <远程地址> (fetch)
  <远程仓库名> <远程地址> (push)
  ```

  第一行为拉取的地址和远程仓库别名，第二行为推送的地址和远程仓库别名

  Git 允许拉取和推送用不同的地址，也允许拉取和推送对应多个地址。

## Idea文件颜色对应Git管理状态

| 文件状态  | 颜色   | 含义                 | 下一步操作   |
| --------- | ------ | -------------------- | ------------ |
| Untracked | 🔴 红色 | 新文件未纳入版本控制 | `git add`    |
| Modified  | 🟦 蓝色 | 修改了旧文件         | `git add`    |
| Staged    | 🟩 绿色 | 已加入暂存区，等提交 | `git commit` |

## 版本控制系统

### 分类

目前最流行的版本控制系统课分为两类：

**集中式版本控制系统**

所有的文件保存在中央服务器上，个人电脑上只保存一个副本。当修改文件时，需要从中央服务器下载最新的版本，然后添加修改内容，再上传回中央服务器。如SVN，CVS

- 使用简单
- 中央服务器的单点故障，会导致所有工作瘫痪。

**分布式版本控制系统**

每个人的电脑上均有完整的版本库。如Git，Mercirial

## Git合并规则

所有开发分支均基于一个统一的Test分支，合并均合并到Test分支

- 在合并前，先拉取远程最新的Test分支，然后在本地合并，解决冲突后，再提交pr

1️⃣ 拉取远程最新的 test 分支

```
git checkout test
git pull origin test
```

#### 2️⃣ 切回自己的功能分支

```
git checkout feature/xxx
```

3️⃣ 将最新的 test 分支合并到当前分支

```
git merge test
```

如果有冲突：

- 在本地解决冲突
- 本地编译 & 测试通过

4️⃣ 提交合并后的代码

```
git add .
git commit
git push origin feature/xxx
```

5️⃣ 再提交 PR 到 test 分支
