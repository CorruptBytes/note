# 策略模式

> 策略模式（Strategy Pattern）定义了一系列可互换的算法(这一系列算法统称为算法族)，将每一个算法封装在独立的策略类中，并使它们可以互相替换，从而使得算法的变化独立于使用算法的客户端。

# 观察者模式

> 观察者模式(Observer Pattern)定义了对象之间的一对多依赖,当对象改变状态时，它的所有依赖者都会收到通知并自动更新。

​	在观察者模式中，被观察的对象称为"主题"(subject)或可观察者(Observavle)，是一个具有"状态"的对象；接收数据的对象称为观察者(observer),是一个需要使用主题"状态"的对象。观察者需要订阅(注册)主题以便在主题数据改变时能够收到更新。主题负责管理数据，并在数据更新时通知已经订阅的观察者。

- 优点
  1. 主题独立管理一份数据，并通知其观察者。相比于让许多对象控制同一份数据，可以得到更干净的OO设计。

## 实现

> 观察者模式有多种实现方法

- Subject与Observer接口

  1. 定义Observer与Subject接口

     自定义Subject与Observer接口规范主题与观察者的行为。

     ```java
     public interface Subject {
         //注册观察者
         public abstract void registerObserver(Observer o);
         //移除观察者
         public abstract void removeObserver(Observer o);
         //数据更新通知观察者
         public abstract void notifyObservers();
     }
     
     public interface Observer {
         public abstract void update(data);
     }
     ```

     > 注意:
     >
     > - 一个具体的主题总是实现`Subject`接口，还可能有设置和获取状态的方法
     > - 观察者可以是实现`Observer`的任意类，观察者必须注册主题才能获取更新

  2. 创建主题，实现Subject接口，且主题内要维护一个观察者列表。

     ```java
     public class WeatherData implements Subject {
        private ArrayList<Observer> observers;
          public WeatherData() {
             this.observers = new ArrayList<>();
         }
          @Override
         public void registerObserver(Observer o) {
             observers.add(o);
         }
         @Override
         public void removeObserver(Observer o) {
             observers.remove(o);
         }
     
         @Override
         public void notifyObservers() {
             for (Observer observer : observers) {
                 observer.update(temperature,humidity,pressure);
             }
         }
     }
     ```

  3. 定义观察者，观察者要维护一个所依赖主题的引用。

     ```java
     public class CurrentConditionsDisplay implements Display, Observer {
         private double temperature;
         private double humidity;
         private Subject weatherData;
     
         public CurrentConditionsDisplay(Subject weatherData) {
             this.weatherData = weatherData;
             weatherData.registerObserver(this);
         }
     
         @Override
         public void update(double temp, double humidity, double pressure) {
             this.temperature = temp;
             this.humidity = humidity;
             display();
         }
     
         @Override
         public void display() {
             System.out.println("Current conditions:" + temperature + "F degrees and " + humidity + "%humidity");
         }
     }
     ```

     > 注意:
     >
     > - 如果后续不需要取消订阅主题，那么观察者中没必要维护一个对主题的引用

- 使用Java内置的Observer与Observable接口

  > 这两个接口在JDK9开始已经被废弃。

## 主题与观察者的数据交互

> 观察者想要获得主题的状态，可以有两种方式:"推"和"拉"。"推"即当主题状态更新时，将状态传入update方法中，此时获取的数据由主题主导，这种观察者模式被称为`推(push)模型`。"拉"即主题提供一些获得状态的getter方法，当状态更新时，主题只需通知观察者数据更新，观察者自行调用getter方法活得想要的数据，这种观察者模式称为`拉(pull)模型`。

- 推模型

  ```
   @Override
      public void update(double temp, double humidity, double pressure) {
          this.temperature = temp;
          this.humidity = humidity;
          display();
      }
  ```

- 拉模型

  ```java
  
  @Override
  public void update(Subject subject, Object data) {
          if (subject instanceof WeatherData) {
             WeatherData weatherData = (WeatherData) subject;
              this.temperature = weatherData.getTemperature();
  
          }
          display();
      }
  ```

> 注意:
>
> - 推与拉不是固定的，在update方法中，主题在通过形参data传递数据的同时，也可以提供getter方法让观察者自行获得数据。
> - 现实中，也可以使用推与拉混合策略，主题推送关键字段的同时，观察者拉取他所需要的其他字段

# 装饰者模式

> 装饰者模式(Decorator Pattern)动态地将责任附加到对象上。若要扩展功能，装饰者提供了比继承更有弹性
> 的替代方案。

- 特点
  - 装饰者和被装饰者对象有相同的超类型
  - 可以用一个或多个装饰者包装一个对象
  - 既然装饰者和被装饰对象有相同的超类型，所以在任何需要原始对象（被包装的）的场合，可以用装饰过的对象代替它。
  - 装饰者可以在所委托被装饰者的行为之前或之后，加上自己的行为，以达到特定的目的。
  - 对象可以在任何时候被装饰，所以可以在运行时动态地、不限量地用你喜欢的装饰者来装饰对象。

## 实现

1. 装饰者与被装饰者都需要实现同一个顶层组件类(或接口)
2. 为区分装饰者与最底层的组件，创建一个装饰者类，需要继承顶层组件类
3. 最底层的组件实现顶层组件类，可作为装饰者的类实现装饰者类(装饰者需要有一个成员变量保存被装饰者的引用，以便后续调用其方法)

> 注意:
>
> - 装饰者模式会导致设计中出现许多小对象，如果过渡使用，会让程序变得很复杂。

# 工厂模式

> 工厂模式是用来封装对象的创建

## 简单工厂

> 简单工厂并不属于工厂模式，而是一种非常实用的编程习惯。简单工厂仅仅是将创建对象的行为委托给工厂，工厂通过不同的条件返回不同的对象

## 工厂方法模式

> Factory Method Pattern，定义了一个创建对象的接口，但由子类决定要实例化的类是哪一个。工厂方法让类把实例化推迟到子类。

- 实现

  > 工厂方法模式一般由创建者类与产品类组成

  1. 定义一系列创建者类族，顶层的抽象创建者类定义一个抽象的工厂方法和一系列依赖于抽象产品的具体方法，这些抽象产品由工厂方法提供，子类实现出具体的工厂方法实例化自己所需的对象。

  2. 定义一系列产品类族，这些产品是工厂方法的具体实例化对象，他们具有一个共同的顶层接口(或类)，便于与创建者解耦

     ![image-20250626102805962](D:\笔记\图片\image-20250626102805962.png)

     > 注意:
     >
     > - 工厂方法模式让子类决定该创建的对象是什么，来达到将对象创建的过程封装的目的。
     > - 工厂方法和创建者并不总是抽象的，可以定义个默认的工厂方法来生产默认的产品，这样即使创建者没有任何子类，依然可以创建产品
     > - 通常情况下，工厂方法只会产生一种对象。根据传入的参数创建不同的对象的工厂方法，称为参数化工厂方法
     > - 简单工厂注重“封装创建”，而工厂方法模式更注重“灵活扩展”，通过多态让系统更易于新增产品、避免修改原有逻辑。

## 抽象工厂模式

> 抽象工厂模式提供一个接口，用于创建相关或依赖对象的家族，而不需要明确指定具体类。
>
> 抽象工厂定义了一个负责创建一组产品的接口。这个接口内的每一个方法都负责创建一个具体产品，利用实现接口的子类来提供具体实现。
>
> 抽象工厂常使用多个工厂方法来实现

![image-20250626130725822](D:\笔记\图片\image-20250626130725822.png)

- 工厂方法与抽象工厂
  1. 工厂方法使用继承：把对象的创建委托给子类，子类实现工厂方法来创建对象。抽象工厂使用对象组合：对象的创建被实现在工厂接口所暴露出来的方法中。
  2. 抽象工厂可以把一群相关的产品集合起来，工厂方法通常只用于创建一个产品

## 静态工厂

> 将工厂类中将创建对象的方法改定义为静态方法，优点是不需要创建工厂对象，缺点是不能通过继承来改变创建方法的行为

# 单例模式

> Singleton Pattern。在这种模式下，一个类只能在程序中只能实例化一个具体对象。这是很有必要的，如线程池(threalpool)，缓存(cache),日志对象，设备的驱动程序对象等。这类对象只能有一个实例，如果制造出多个实例，可能会导致程序的行为异常，资源使用过量等问题

- 定义

  > 单件模式确保一个类只有一个实例，并提供一个全局访问点

## 实现

> 最简单的单件模式实现通过将构造方法私有，提供getInstance()方法返回对象

```java
public class Singleton {
    private static Singleton singleton;
    private Singleton() {

    }
    public static Singleton getInstance() {
        if (singleton == null) {
           singleton = new Singleton();
        }
        return singleton;
    }

}
```

> 注意:
>
> - 此时的单件模式无法保证在多线程内单件

### 对于多线程的解决方案

- 在getInstance()方法上加入synchronized关键字使其变成同步代码

  ```Java
  public static synchronized ChocolateBoiler getInstance() {
          if (chocolateBoiler == null) {
             chocolateBoiler = new ChocolateBoiler();
          }
          return chocolateBoiler;
      }
  ```

  > 只有第一次执行getInstance()方法时，才需要同步，只要静态变量被设置好，后续调用都不需要同步，因此同步在后续调用会变成一个累赘。如果不是性能敏感的场合，可以这样做

- 直接通过静态变量加载实例，不使用延迟实例化的做法

  ```java
  public class Singleton {
  	private static Singleton uniqueInstance = new Singleton();
  	private Singleton() {
  	
  	}
  	public static Singleton getInstance() {
  		return uniqueInstance;
  	}
  	}
  ```

  > 如果创建一个实例的负担不太沉重，并且应用程序总是会创建并使用单件实例，可以采取这种方案

- 通过双重检查锁机制(double-checked locking)，在getInstance()方法中减少使用同步，称为DCL单例模式

  - 优点是懒惰实例化，且只有第一次使用时才需要加锁，后续使用无需加锁。
  
  ```
  public class Singleton {
  	
      private static volatile Singleton singleton;
  
      private Singleton() {
          
      }
      public static Singleton getInstance(){
          if (singleton == null) {
          sychronized(Singleton.class) {
          	if (singleton == null) {
          	singleton = new Singleton();
          	}
          }
          
          }
          return singleton;
      }
  }
  ```
  
  - `volatile`关键字是为了防止JVM进行指令重排优化，导致在创建单例时，先对静态变量赋值，再调用构造方式，使得某个线程获得了一个未初始化的对象。

# 命令模式

> 命令模式可将"动作的请求者"从"动作的执行者"对象中解耦，把请求封装成一个命令对象，当请求到来时，委托命令对象做相关的动作。请求者不必知道动作是什么，只要有一个命令对象能和正确的对象沟通，完成动作即可。

- 定义

  > 命令模式将"请求"封装成对象，以便使用不同的请求，队列或者日志来参数化其他对象，命令模式也支持可以撤销的操作

  ![image-20250628102445940](D:\笔记\图片\image-20250628102445940.png)

## 实现

1. 让所有的命令对象都实现相同的包含一个方法的接口，也就是命令接口，其中的方法是调用命令的入口。Command接口也可以提供撤销命令的uodo方法，在undo方法中，只需要做出与execute方法相反的动作即可

   ```
   public interface Command {
   	public void execute();
   }
   ```

2. 实现真正的命令类，这个类要实现Command接口，并且要封装动作执行者的引用以供在excute方法中执行动作

   ```java
   public class LightCommand implements Command {
       private Light light;
       private LightCommand(Light light) {
           this.light = light;
       }
       @Override
       public void execute() {
           light.on();
       }
   }
   ```

3. 想要使用命令时，可将命令对象作为成员变量存储在委托类中，并通过settter方法设置命令，当需要调用命令时，只需要调用命令的execute方法即可

   ```java
   public class SimpleRemoteControl {
      private Command slot;
      public SimpleRemoteControl(){}
       public void setCommand(Command command) {
          this.slot = command;
       }
       public void buttonPress() {
          slot.execute();
       }
   }
   ```

## 宏命令

> 在宏命令中，通过数组存储一大堆命令对象，当执行宏命令时，一次性调用数组中的所有命令。一般来说，宏命令所执行的命令序列可以动态改变而不是硬编码

## "聪明"命令对象与"傻瓜"命令对象

> 聪明命令对象直接实现请求，傻瓜命令对象将工作委托给接收者，这两种命令对象都有使用场景

# 适配器模式

> 适配器模式将一个类的接口，转换成客户期望的另一个接口。适配器让原本接口不兼容的类可以合作无间。

## 适配器

> "适配器"将一种接口转换成另一种接口

1. 适配器类需要实现目标接口

   ```
   public class TurkeyAdapter implements Duck
   ```

2. 适配器类需要存储一个被适配对象的引用

   ```
   public class TurkeyAdapter implements Duck{
       private Turkey turkey;
   
       public TurkeyAdapter(Turkey turkey) {
           this.turkey = turkey;
       }
   }
   ```

3. 在适配器类中调用适配对象的方法实现所需接口的功能

   ```
   public class TurkeyAdapter implements Duck{
       private Turkey turkey;
   
       public TurkeyAdapter(Turkey turkey) {
           this.turkey = turkey;
       }
   
       @Override
       public void fly() {
           turkey.fly();
       }
   
       @Override
       public void quack() {
           turkey.gobble();
       }
   }
   ```

![image-20250630102737277](D:\笔记\图片\image-20250630102737277.png)

> 注意:
>
> - 这种实现被称为对象适配器，他通过组合的方式将客户的请求委托给被适配者
> - 还有一种适配器被称为类适配器，他通过继承来实现请求委托，但由于类适配器没有很好的弹性，一般很少使用。(类适配器一般使用多继承实现，但由于Java不支持多继承，在Java中，一般采用实现目标接口，继承被适配者的方式实现)

![image-20250630103325826](D:\笔记\图片\image-20250630103325826.png)

- 类适配器 vs 对象适配器
  1. 对象适配器使用组合，可以适配某个类和其子类，类适配器使用继承，只能适配某个特定类
  2. 类适配器继承被适配类的功能，无需重新实现整个被适配者
  3. 使用类适配器时，仅仅需要一个类适配器；使用对象适配器时，需要一个对象适配器和一个被适配者

# 外观模式

> 外观模式(Facade-Pattern)提供了一个统一的接口，用来访问子系统中的一群接口。外观定义了一个高层接口，让子系统更容易使用。

- 外观模式 vs 适配器模式 vs 装饰者模式

  > 适配器模式将一个接口转成另一个接口，外观模式同样改变接口，但是目的是简化接口，装饰者模式不改变接口，只是加入新的功能。

# 模板方法模式

> 模板方法模式定义了一个算法的**骨架（模板）**，将一些步骤延迟到子类中实现。这样，子类可以在不改变算法整体结构的前提下，重新定义算法的某些步骤。

- 实现

  > 模板就是一个方法，这个方法将算法定义为一组步骤，其中任何步骤都可以是抽象的，由子类负责实现，这可以保证算法的结构保持不变，同时由子类提供部分实现

  ![image-20250701192308001](D:\笔记\图片\image-20250701192308001.png)

  > 注意:
  >
  > - 模板方法一般会用final修饰，以防止子类修改算法的结构
  > - 工厂方法是一个特殊的模板方法，它专门用于创建并返回对象。

## 钩子

> 在模板方法中，可以有这样一种方法:它默认是一个空方法或者由抽象类提供默认实现，子类可以视情况决定是否覆盖它。这种方法被称为hook(钩子)。钩子让子类有能力对算法的不同点进行挂钩。

## 远程代理

> 远程代理就是远程对象的本地代表，远程对象是在不同Java虚拟机中的对象。通过一个本地代表对象，调用本地方法将其行为转发到远程对象中。

- RMI

  > Java提供的远程代理的一种方法，在RMI中，客户辅助对象称为stub(桩),服务辅助对象称为skeleton(骨架)

  1. 定义远程接口，这个接口定义了客户可以远程调用的方法。stub与实际的服务对象都要实现此接口
  2. 实现提供远程服务的类
  3. 利用rmic产生stub和skeleton
  4. 启动RMI register，客户可以从中查到代理的位置
  5. 实例化一个服务对象，并将其注册到RMI register。之后，这个服务就可以供客户调用

## 动态代理

> 是一种在 **运行时** 动态生成代理对象的机制，可以在不修改目标类代码的情况下，对方法调用进行拦截、增强或替换行为。

- 动态代理会在运行期生成代理类的字节码，只是默认不会落盘成 `.class` 文件。

常见的实现动态代理的方式有两种：①使用JDK所提供的API实现②使用CGLIB库实现

### JDK实现

JDK提供了`java.lang.reflect.Proxy类`和`java.lang.reflect.InvocationHandler`接口实现动态代理

```java
//接口，定义需要被代理的方法
public interface Star {
    public void sing(String name);
}
//被代理的类，实现接口
public class BigStar implements Star {
    private String name;
    @Override
    public void sing(String name) {
        System.out.println("唱歌");
    }
}
    public static Star createProxy(BigStar bigStar) {
        /**
         * 参数一:指定使用的类加载器，它会加载生成的代理类
         * 参数二：指定接口，这个接口指定了代理类所具有的方法，可以同时指定多个接口
         * 参数三：InvocationHandler对象，定义代理的增强逻辑,可以被多个代理类复用
         * 返回值：代理对象
         */
        return (Star) Proxy.newProxyInstance(
                ProxyUtil.class.getClassLoader(),
                new Class[]{Star.class},
                /*代理对象调用所有方法时都将委托给 InvocationHandler 的 invoke 方法，在 invoke 中可以在调用目标方法前后添加增强逻辑*/
                new InvocationHandler() {
                    @Override
                    /**
                     * 参数一：生成的代理对象
                     * 参数二：被调用的方法对象
                     * 参数三：被调用方法的参数
                     * 返回值：此次调用的返回值
                     */
                    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
                        if ("sing".equals(method.getName())) {
                            System.out.println("唱歌前的准备");
                        }
                        return method.invoke(bigStar,args);
                    }
                });
    }
```

# 生成器模式

生成器模式(`Builder Pattern`)是一种用于构建复杂对象的设计模式，它将对象的构建过程封装起来，并允许按步骤构造对象，使同一构建过程可以创建不同的对象表示。

<h4>优点</h4>

- **提高代码可读性：**通过链式调用，逐步构造对象，方法名直接表示参数含义。
- **解决构造方法参数爆炸：**复杂对象构造方法需要多个参数，且需要重载多个构造方法。使用生成器构造对象时，对外只暴露 `Builder`，不再需要多个重载构造方法，对象内部只保留一个由 `Builder` 调用的构造入口。
- **集中校验，保证对象合法性：**校验逻辑集中在 `build()`中
- **对扩展友好：**新增字段时，只需给 Builder 加一个方法，无需修改大量构造方法。

## 示例

- **将目标类的构造方法私有化**，限制对象只能通过 Builder 创建，防止外部直接调用构造方法。
- **提供静态内部类 `Builder`**，用于临时保存构建对象所需的各个字段，并为每个字段提供对应的`setter`方法；在 `build()` 方法中统一进行参数校验，并完成对象的构造。

```java
public class User {
    private final String name;
    private final int age;
    private final String email;
    // 构造方法对外隐藏，只能由 Builder 调用
    private User(Builder builder) {
        this.name = builder.name;
        this.age = builder.age;
        this.email = builder.email;
    }
    public static class Builder {
        private String name;     // 必填
        private int age;         // 可选
        private String email;    // 可选

        public Builder name(String name) {
            this.name = name;
            return this;
        }
        public Builder age(int age) {
            this.age = age;
            return this;
        }
        public Builder email(String email) {
            this.email = email;
            return this;
        }
        public User build() {
            // 统一校验
            if (name == null || name.isEmpty()) {
                throw new IllegalStateException("name is required");
            }
            return new User(this);
        }
    }
}
```

# 责任链模式

责任链模式是一种 **行为型设计模式**，它将多个处理对象串成一条链，请求沿着链依次传递，每个处理器可以选择处理请求或将其传递给下一个

- 解耦请求发送者和处理者:调用者不关心谁处理,只需要将其放入责任链中即可。

- 通过改变链内的成员或调动次序，可以动态地新增或删除责任。

<h3>实现模板</h3>

```java
//代表一个处理器
abstract class Handler {
    protected Handler next;
    public void setNext(Handler next);
    public abstract void handle(Request request);
}
//其中`handle`方法的核心逻辑为
if (能处理) {
    处理;
} else if (next != null) {
    next.handle(request);
}
```

<h3>类型</h3>

常见的责任链有两种

- 纯责任链：只要有一个处理成功，就停止。
- 增强型责任链：责任链中的每个节点都会执行。

## 设计原则

- 找出程序中会变化的方面，并将其与不变的方面分离。或者说把会变化的部分取出并封装起来，以便以后可以轻易改动或扩充此部分。这会导致代码变化引起的不经意后果变少，系统变得更有弹性。
- 针对接口编程，而不是针对实现编程。这里的接口指的是超类型(如Java中的抽象类或接口)
- 多用组合，少用继承。可以将一个类与其他对象组合实现功能，如将一个类的行为委托给其他对象，而不是自己实现。这可以使系统具有弹性，也可以提供在运行时动态地改变行为的可能。
- 为了交互对象之间的松耦合设计而努力。本质是将对象之间的互相依赖降低。如观察者模式中主题与观察者之间唯一依赖的东西是一个实现Observer接口的列表。只要遵循主题与观察者之间的接口，就可以自由的改变主题或观察者。
- 开放-关闭原则:类应该对扩展开放，对修改关闭。当需要修改功能时，应该通过添加新代码来实现，而不是修改已经稳定工作的旧代码。满足开闭原则的系统：应允许通过**新增代码**（而不是修改原代码）来实现新功能。一旦某部分模块稳定运行，**不应该为了加功能去改它的内部实现**，防止引入新 bug。
- 依赖倒置原则(Dependency Inversion Principle):要依赖抽象，不要依赖具体类。“倒置”就是让高层模块依赖“稳定的抽象”而非“多变的细节”，并让细节去适配抽象，而不是反过来。要遵循以下方针:①变量不可以持有具体类的引用②不要让类派生自具体类③不要覆盖基类中已实现的方法。
- 最少知识原则:只和你的密友谈话。要遵循在对象的方法内，我们只该调用属于以下范围的方法:①该对象本身
  ②被当做方法的参数而传递进来的对象③此方法所创建或实例化的任何对象④对象的任何组件
- 好莱坞原则:
- 单一责任原则:一个类应该只干一件事，类的每个责任都有改变的潜在区域。超过一个责任，意味着超过一个改变的区域，应该保证一个类应该只有一个引起变化的原因。内聚(cohesion）这个术语你应该听过，它用来度量一个类或模块紧密地达到单一目的或责任。当一个模块或一个类被设计成只支持一组相关的功能时，我们说它具有高内聚，反之，当被设计成支持一组不相关的功能时，我们说它具有低内聚。内聚是一个比单一责任原则更普遍的概念，但两者其实关系是很密切的。遵守这个原则的类容易具有很高的凝聚力，而且比背负许多责任的低内聚类更容易维护。

## 注册表

> 一种用于集中管理、查找、调度某一类对象（通常是行为、服务或组件）的容器。

- 特性

  |          特性          |                    说明                    |
  | :--------------------: | :----------------------------------------: |
  |      **映射能力**      | 把 key（字符串、类型、ID）映射到对象或行为 |
  |     **集中式管理**     |   所有策略或模块都集中注册，避免逻辑分散   |
  |   **解耦查找与实现**   |    使用者只管调用注册表，不用知道实现类    |
  | **支持扩展和动态注册** |        可动态添加、移除、替换注册项        |

- 实现

  > 通常采用哈希表实现，通过键值对存储并查找表项

  ```java
  Map<String, OperationStrategy> map = new HashMap<>();
  //将策略注册进注册表中，通过key来一一映射调用相应的策略。
  map.put("+", (a, b) -> a + b);
  map.put("-", (a, b) -> a - b);
  map.put("*", (a, b) -> a * b);
  map.put("/", (a, b) -> a / b);
  ```

  

# 代理

代理的目的是转移主类的部分职责给其他类，保证主类功能单一明确。通过代理，可以无侵入式的为代码增加额外的功能

- 特点
  1. 对象有什么方法，代理就一定要有对应的方法。

- 实现

> 代理与被代理的对象需要实现同一个接口，接口中定义需要被代理的方法，在代理中调用被代理对象所实现的方法，此时，如果想要调用功能，直接找代理即可

1. 创建一个接口，这个接口规定了需要代理的方法，代理类与主类都需要实现这个接口。

   ```java
   public interface Star {
       public abstract String sing(String singName);
       public abstract void dance();
   }
   
   public class BigStar implements Star{
       private String name;
   
       public BigStar() {
       }
   
       public BigStar(String name) {
           this.name = name;
       }
       @Override
       public String sing(String singName) {
           System.out.println(this.name + "正在唱" + singName);
           return "谢谢";
       }
       @Override
       public void dance() {
           System.out.println(this.name + "正在跳舞");
       }
   ```

2. 创建代理对象

   Java中通过java.lang.reflect.Proxy类的newProxyInstance（)方法创建代理对象

   //参数三：用来指定生成的代理对象要干什么事情
