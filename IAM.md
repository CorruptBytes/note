# 概述

`IAM`(身份与访问管理,`Identity and Access Management`)是一套用于 **管理“谁可以访问什么资源，以及如何访问”** 的系统。`IAM`通常由以下几个核心能力:

- **身份管理**：负责管理用户身份,包括确保用户在系统中拥有唯一身份标识；身份声明周期管理,自动化用户账户的创建、修改、禁用和删除流程；身份同步与集成：将用户身份数据从各种来源中同步到统一的IAM平台并分发给下游应用
- **认证管理:**包括用户身份认证，支持多种认证方式，单点登录(SSO)等功能。
- **权限管理与访问控制**：负责控制用户可以访问哪些资源和执行哪些操作，例如角色权限管理、RBAC/ABAC、资源访问控制等。
- **审计：**详细记录所有与身份、认证、授权相关的操作
- **应用管理：**管理接入IAM系统的应用信息，简化新应用的接入。



# 概述

`SSO(Single Sign-On)`即单点登录，是一种 **身份认证机制**，允许用户在多个应用系统中，只需要登录一次就可以访问所有相互信任的应用系统，而无需在每个应用中重复登录。

`SSO`的核心就是建立一个统一的身份认证中心，让多个系统共享同一个登录状态，实现一次登录访问多个系统。

## 应用场景

- **企业内部系统**
   企业通常有多个系统，如 OA、ERP、运维系统等。通过 SSO，用户在一次登录后即可访问所有受信任的企业系统，无需重复登录，真正实现“一次登录，多系统访问”。
- **第三方登录**
   例如使用微信、Google、GitHub 账号登录第三方网站。
  - 严格意义上，这不完全符合 SSO 的“一次登录访问所有受信任系统”定义，因为跨不同网站仍需授权。
  - 但它在实现架构上与`SSO`很相似，且借助统一身份源，实现了类似 SSO 的理念：用户只需一个账号，就能在每个网站内快速登录，享受单点登录体验。

## 优点

- 用户体验好
- 安全性更高，账号密码 **只在认证中心输入一次**。
- 统一权限管理
- 易扩展

# SSO基本架构

```
			用户浏览器
      			│
      			▼
		 +------------------+
         |   	IDP
         	 统一认证中心     |
         |  (Auth Server)   |
         +--------+---------+
                  |
        认证成功返回Token
                  |
   +--------------+--------------+----------------+
   |                             |				  | 	
+--v----+                  +-----v----+			  |	
|系统A  |                  |系统B     |			   
|SP    |                  |SP        |			...更多系统
+-------+                  +----------+
```

<h3><code>IDP</code></h3>

`Identity Provider`,专门负责用户认证并颁发凭证的系统，是`SSO`中的核心组件。

## 原理

`SSO`的核心就是两件事：

- 统一认证中心
- 共享登录状态

当系统接入`SSO`后，其登录流程为

1. 系统访问 → 重定向 SSO 验证
2. 登录成功 → SSO创建全局登录 Session，并写入 Cookie
3. SSO 向目标系统颁发 Token，重定向回目标系统，此后系统可以根据此`Token`向SSO系统获取登录用户信息
   - `SSO`颁发`Token`的主要意义作为一次性或短期凭证，告诉目标系统“该用户已通过 SSO 登录验证”，而不是通过此`Token`管理用户登录状态。
4. 系统创建本地`Session`管理用户登录状态。
5. 如有其他被信任系统被访问，会重定向到`SSO`验证界面，且携带全局`SessionId`，`SSO`通过`SessionId`获取用户登录状态，无需再次登录，直接向目标系统颁发`Token`。

## `OAuth2`

`OAuth2`(开放授权协议)是一个授权协议，核心是让第三方应用在不获取用户密码的情况下访问用户资源。

`OAuth2`定义了在授权过程中的四个核心角色：

|          角色          |        含义        |
| :--------------------: | :----------------: |
|    `Resource Owner`    | 资源拥有者（用户） |
|        `Client`        |     第三方应用     |
| `Authorization Server` |     授权服务器     |
|   `Resource Server`    |     资源服务器     |

四个核心角色的关系如下

```
用户
 │
 │授权
 ▼
第三方应用(Client)
 │
 │向授权服务器申请Token
 ▼
授权服务器
 │
 │返回AccessToken
 ▼
第三方应用
 │
 │拿Token访问
 ▼
资源服务器
```

### 四种授权模式

OAuth2 定义了 **四种授权方式**。

<h4><code>Authorization Code</code></h4>

## `OIDC`

**OIDC（OpenID Connect）** 是一种建立在 OAuth2 之上的 **身份认证协议**，通过在 OAuth2 授权协议之上增加身份层（Identity Layer），为客户端提供标准化的用户身份认证能力。

### 核心概念

**`ID Token`**

类型固定为`JWT`，用于标识用户身份，告诉客户端用户是谁。

**`Access Token`**

类型不定，用于访问受保护的资源 API。

**`Refresh Token`**

类型不定，用于刷新 `Access Token`。可以在刷新`Access Token`的同时刷新 `Refresh Token`，实现重复刷新；也可以使用刷新一次即失效的策略，防止被盗用重复使用。

**`Client`**

需要用户登录的应用（SP，Service Provider）,也就是需要用户认证的业务系统。

**`Authorization Server / IdP`**

身份提供者，进行发放 Token，管理用户认证。

**`Scopes`**

为一个字符串，标识请求的权限范围（`openid`, `profile`, `email` 等）

**`claims`**

ID Token 中的用户信息，例如 `sub`, `email`, `name`

### 认证流程

应用(`Client`)想要使用`OIDC`实现用户登录前，需要先在`IdP`上注册应用，以获取每一个应用独属的 `client_id` 和 `client_secret`，并配置回调地址 `redirect_uri`

1. 用户访问应用，但未登录，重定向到 **OIDC 授权端点**，也就是统一的认证界面：

   ```http
   GET https://idp.example.com/authorize
   ?response_type=code
   &client_id=your_client_id
   &redirect_uri=https://yourapp.com/callback
   &scope=openid|profile|email
   &state=xyz
   ```

   - `response_type`：授权模式，`code`为授权码模式（推荐）
   - `scope`：标识请求的权限范围,`openid`表示请求 OIDC 登录
   - `state`：防止CSRF

2. 用户在 IdP 登录（输入用户名密码或二步验证），IdP 验证通过后，生成 **授权码（Authorization Code）**，重定向回应用的 `redirect_uri`，携带 `code` 和 `state`。

   ```
   https://yourapp.com/callback?code=abc123&state=xyz
   ```

3. 客户端用授权码换 Token

   ```
   POST https://idp.example.com/token
   Content-Type: application/x-www-form-urlencoded
   
   grant_type=authorization_code
   code=abc123
   redirect_uri=https://yourapp.com/callback
   client_id=your_client_id
   client_secret=your_client_secret
   IdP返回
   {
     "access_token": "xxx",
     "id_token": "yyy", 
     "refresh_token": "zzz",
     "expires_in": 3600,
     "token_type": "Bearer"
   }
   ```

   - **id_token** → 用来标识用户身份（JWT）
   - **access_token** → 用于访问受保护 API
   - **refresh_token** → 用于刷新 access_token

4. 应用验证`ID Token`:

   - 检查签名是否正确（通过 IdP 的公钥）
   - 检查 `aud` 是否匹配你的客户端
   - 检查 `exp` 是否过期
   - 可从 `claims` 获取用户信息

5. 应用生成本地 登录标识(`Session`或`JWT`等)

# 加密算法/签名算法

加密算法用于保密数据，使只有授权方能读；签名算法用于证明数据来源和完整性，防止篡改

## RSA

`RSA`是一种 **非对称加密算法**。它使用一对密钥，既可以用于数据加密/解密，也可用于数字签名/认证。

- 用于数据加密/解密时，公钥加密， 私钥解密。
- 用于数字签名/认证时，私钥签名，公钥认证。

`RSA`的安全性依赖于大整数分解的困难性。

# 密码保护策略

现代密码存储和防护的核心策略是永远不要存储用户密码的明文，而是存储密码的哈希值

### 常见哈希算法

哈希函数是一个将任意长度输入映射为固定长度输出的函数。对于密码学哈希函数，还要求其单向性。

## 权限模型

权限模型是系统中定义用户在什么条件下可以访问什么资源、执行什么操作的规则体系。

大多数权限模型都包含四个部分：

|         概念         |   含义   |       示例       |
| :------------------: | :------: | :--------------: |
| **Subject（主体）**  | 谁在操作 |  用户、系统账号  |
| **Resource（资源）** | 操作对象 | 订单、商品、用户 |
|  **Action（操作）**  | 操作类型 | 查看、创建、删除 |
|  **Policy（策略）**  | 访问规则 |   谁可以做什么   |

<h3>常见的权限模型</h3>

<h4><code>RBAC</code></h4>

基于角色的权限控制(`Role Based Access Control`)，核心思想是权限 **不直接分配给用户**，而是分配给角色，用户通过角色获取权限。

**优点**

- 简单
- 易管理

**缺点**

- 不够灵活

**主要表**

- 用户（User）
- 角色（Role）
- 权限（Permission）
- 用户角色关联表
- 角色权限关联表

<h4><code>ABAC</code></h4>

基于属性的权限控制(`Attribute Based Access Control`)，核心思想是权限由 **属性规则**决定，基于不同的属性条件为赋予权限

**典型属性**

- 用户属性（department、level、region）
- 资源属性（owner、type）
- 环境属性（时间、IP、设备）

**优点**

- 非常灵活
- 适合复杂系统

**缺点**

- 实现复杂
- 规则难管理

------

绝大多数企业系统采用 **RBAC + ABAC** 的混合权限模型, 其中 **RBAC 用于定义用户的功能权限（能做什么操作）**， **ABAC 用于根据属性规则限制用户可访问的数据范围（能操作哪些数据）**。

<h4><code>PBAC</code></h4>

基于策略的访问控制（PBAC - Policy-Based Access Control），核心思想是通过定义一系列策略规则来决定访问决策，通常与ABAC结合使用

## CSRF

**Cross-Site Request Forgery** 的缩写,意为跨站请求伪造，是网络安全中常见的一种攻击方式。

本质是**攻击者利用用户已经登录过的身份，在用户不知情的情况下，让用户浏览器向受信任网站发送恶意请求**，从而执行一些未授权的操作。

**特点**

- 用户已经登录目标网站（有有效的 Cookie 或 Session）。
- 攻击发生在用户不知情的情况下。
- 攻击利用了浏览器自动携带 Cookie 的机制。

## XSS

**XSS（Cross-Site Scripting，跨站脚本攻击）** 是一种常见的 Web 安全漏洞，攻击者通过向网页中注入恶意脚本（通常是 JavaScript），让这些脚本在其他用户的浏览器中执行。

- `XSS`与`SQL`注入一样，都属于一种注入攻击，本质原因是系统未能正确区分数据和代码

<h3><code>XSS</code>的危害</h3>

攻击者可以通过注入的脚本执行任意前端操作，如：

- 窃取`cookie`
- 冒充用户发起请求
- 窃取页面数据
- 伪造登录框钓鱼

### `XSS`类型

<h4>存储型XSS</h4>

恶意脚本被**存入服务器（数据库）**，所有访问该数据的用户都会被攻击。

<h4>反射型XSS</h4>

恶意代码通过 请求参数 传入，服务器“原样返回”，浏览器执行。

<h4>DOM型XSS</h4>

**完全发生在前端的跨站脚本攻击**，前端 JS 直接使用用户可控的数据修改页面 DOM，导致恶意脚本在浏览器执行的一种攻击

```
URL参数 → 前端JS处理 → 插入DOM → 执行
```

## `IDaas`

`Idaas`即`Identity as a  service`(身份即服务)，是一种把"用户身份认证和权限管理"放到云端提供的服务模式。

传统模式下，公司在内部搭建`IAM`平台以进行统一的身份认证和权限管理。随着云计算的普及，`IAM`逐渐演进出了“身份即服务”（Identity as a Service, IDaaS）这种交付模式。IDaaS将IAM能力以云服务的形式提供给企业，相比自建 IAM 系统，IDaaS 具有部署成本低、运维简单、扩展性强等优势，使企业能够更高效地构建统一身份认证体系。

## 云计算

云计算是一种通过互联网，以按需、弹性、按量计费的方式，提供远程计算资源及相关服务的技术模式；而云服务是在这种模式下对外提供的具体功能或产品。

## 最小权限原则

最小权限原则（Principle of Least Privilege, `PoLP`）是信息安全和身份管理中最核心的安全原则之一，即用户或程序在系统中只被授予完成其工作所必需的最少权限。

## 彩虹表

彩虹表是一种预先计算好的哈希值表，用于通过反查哈希值快速破解明文密码。

<h4>原理</h4>

系统在存储用户密码时通常会经过哈希处理，哈希函数单向不可逆，通过哈希值很难直接反推出原密码。彩虹表会提前计算大量常用密码及其哈希值，并保存为一个表格。当拿到哈希值时，可以通过彩虹表快速反查出密码。

彩虹表采用链式哈希技术压缩存储空间，每个链头到链尾都可以计算多个哈希值。

本质上，彩虹表就是使用特定压缩技术存储的密码哈希映射表。
