# 概述

## 进程/线程

**进程**

**进程**可以视为一个**程序运行的实例**，是操作系统管理资源的容器。它提供一个封装环境，用来加载指令、分配和管理内存、管理 I/O 等资源。

- 当程序被执行时，操作系统会将程序的代码和数据从磁盘加载到内存，并为其分配所需资源，这一刻就开启了一个进程
- 有的程序可以同时运行多个实例进程(如记事本，浏览器)，有的程序只能运行一个实例进程(如网易云音乐)
- 进程主要用于管理共享资源，供其内部的线程共享使用

**线程**

一个进程内可以存在多个线程，每一个线程都是一个指令流，操作系统会将指令流中的一条条指令以一定的顺序交给CPU执行。

- 线程是操作系统中调度和执行的最小单位，进程是操作系统中资源分配的基本单位。
- Windows中进程不直接运行，只是作为资源的容器，真正执行的是进程里的线程。
- 相比于进程，线程更加轻量，线程上下文切换成本一般比线程上下文切换成本低
- 操作系统中有一个组件叫做任务调度器，它负责调度线程交给CPU执行

## 并发/并行

**并发**

> 对于单核CPU，任务调度器会将CPU的时间片(Windows下时间片最小约为15ms)分给不同的线程使用,由于CPU在线程间切换非常快，宏观上表现为不同线程并行执行，但微观上线程仍然是串行执行的。将这种线程轮流使用CPU的做法称为**并发**(concurrent)。

**并行**

> 对于多核CPU，每一个核心都可以独立执行线程,多个核心可以同时执行多个线程，此时线程在微观上也是并行执行的。

**一般来说，在操作系统中并发与并行是同时存在的，因为现代CPU一般是多核，但是操作系统中的线程远远多于核心数**

## 同步/异步

**同步从不同角度解读具有不同的含义**

- 从方法调用的角度

  **需要等待方法结果返回才继续执行程序**

- 从多线程的角度

  **同步是指在多线程环境中，通过协调线程对共享资源的访问方式，使并发操作有序进行，从而避免数据竞争和并发冲突。**
  
- 从IO的角度

  **发起 IO 操作后，必须等待操作完成，才能继续往下执行。**

------

**异步从不同角度将具有不同的含义**

- 从方法调用的角度

  **不需要等待方法返回结果，调用方法后继续执行程序，通过回调函数等机制在方法完成后对方法结果进行处理**

------

**通过多线程可以将方法执行变为异步，将费时的方法放入新线程中执行，主线程继续执行指令，无需等待方法执行完毕**

# `Thread`

在 Java 中，线程被抽象为 `Thread` 类，这是一个 **JVM 层面的线程抽象**。当我们创建并启动一个 `Thread` 对象时，JVM 会在底层将其映射为一个 **操作系统级线程**。

**特点**

- `Thread` 类中提供的许多方法（如 `start()`、`sleep()`、`interrupt()`、`join()` 等）对底层 **操作系统线程 API 进行封装或间接调用**。这种方式屏蔽了不同操作系统之间的实现差异，让开发者能够以统一的方式使用线程，而无需直接操作操作系统提供的线程接口。

## 线程状态

**从操作系统线程层面来说,线程具有5种状态**

<img src="D:\笔记\图片\image-20251004221548131.png" alt="image-20251004221548131" style="zoom: 50%;" />

- 【初始状态】仅是在语言层面创建了线程对象，还未与操作系统线程关联
- 【可运行状态】（就绪状态）指该线程已经被创建（与操作系统线程关联），可以由 CPU 调度执行
- 【运行状态】指获取了 CPU 时间片运行中的状态
  - 当 CPU 时间片用完，会从【运行状态】转换至【可运行状态】，会导致线程的上下文切换
- 【阻塞状态】
    - 如果调用了阻塞 API，如 BIO 读写文件，这时该线程实际不会用到 CPU，会导致线程上下文切换，进入【阻塞状态】
    - 等 BIO 操作完毕，会由操作系统唤醒阻塞的线程，转换至【可运行状态】
    - 与【可运行状态】的区别是，对【阻塞状态】的线程来说只要它们一直不唤醒，调度器就一直不会考虑调度它们
- 【终止状态】表示线程已经执行完毕，生命周期已经结束，不会再转换为其它状态

**从Java线程层面来讲，`Thread`中的`Thread.State`枚举类，定义了六种线程状态**

<img src="D:\笔记\图片\image-20251004222656532.png" alt="image-20251004222656532" style="zoom:50%;" />

**`NEW `**

初始状态，处于此状态的线程对象刚被创建，但是还没有调用 `start() `方法，也就是还没有与操作系统线程关联。

**`RUNNABLE`** 

可运行状态。调用 `start()` 方法之后，线程对象进入`RUNNABLE`状态，并且与操作系统线程关联。处于 `RUNNABLE `状态的线程可能正在执行，也可能正在等待操作系统的某些资源(如CPU或IO)

**`BLOCKED `**

阻塞状态，处于 `BLOCKED` 状态的线程正等待监视器锁的释放以进入同步区

**`WAITING `**

等待状态，处于`WAITING`状态的线程正在等待其他线程执行某个特定的操作。

以下三个方法会使线程进入`WAITTING`状态：

- `Object.wait()`
- `Thread.join()`
- `LockSupport.park()`

**`TIMED_WAITING `**

超时等待状态，处于`TIMED_WAITING`状态的线程正在等待其他线程执行特定动作，但指定了最大等待时间。如果到达最大等待时间，则该线程会自动唤醒。

以下方法会使线程进入超时等待状态：

- `Thread.sleep(long millis)`
- `Object.wait(long timeout)`
- `Thread.join(long millis)`
- `LockSupport.parkNanos(long nanos)`：
- `LockSupport.parkUntil(long deadline)`

**`TERMINATED`** 

终止状态，处于`TERMINATED`状态的线程已经完全执行完毕。

## 线程方法

### 静态工具方法

<h3>
    <code>sleep</code>
</h3>

```
static void sleep(long millis)
```

- 使当前线程进入`TIMED_WATTING`状态，进入有限时的休眠
- 其他线程可以使用`interrupted`方法中断正在睡眠的线程，此时`sleep()`会抛出`InterruptedException`异常
- 睡眠结束后线程不一定会立刻执行，需要等待操作系统调度。
- 建议使用`TimeUnit.XXX.sleep()`代替`sleep`，可读性更强
- `sleep`常用于某些死循环程序(如服务器),在死循环中加入`sleep`,防止死循环循环过快占用过多CPU资源

<h4><code>Thread.sleep(0)</code>有什么意义</h4>

`Thread.sleep`底层调用操作系统的`sleep`或`nanosleep`系统调用，正常情况下，操作系统会将线程置为休眠状态，当前线程让出CPU的执行权，且不再参与 CPU 竞争。操作系统调度器会再次调度选择一个线程执行。同时操作系统会设置一个定时器，定时器到达时间后唤醒线程。

但是调用`Thread.sleep(0)`时，不会产生实际的睡眠时间，也不会设置定时器唤醒。当前线程会主动让出一次 CPU 执行机会，触发线程调度。操作系统可能会调度一个优先级更高的线程执行，也可能会继续调度当前线程执行。

<h4><code>yield</code></h4>

```java
static native void yield();
```

- 当前线程主动让出 CPU 执行权，提示调度器优先调度其他可运行线程，但不做任何保证。
- 线程调用`yield`前后始终是`RUNNABLE`状态，仍然参与 CPU 竞争，是否发生线程切换完全取决于调度器。

<h4><code>interrupted</code></h4>

```java
static boolean interrupted()
```

- 判断当前线程是否被中断，在调用后会清除中断标记

### 线程实例方法

<h3><code>run</code></h3>

```
void run()
```

- `run()` 方法对应 OS 线程的命令流,是线程真正的执行逻辑
- 如果在初始化` Thread `对象时传递了 `Runnable` 参数，则线程启动后会调用` Runnable `中的 `run()`，否则默认不执行任何操作。还可以创建 `Thread` 的子类对象，重写`run()`方法来覆盖默认行为
- 如果直接调用`run()`，会直接由当前线程执行`run()`，而不是启动一个新的线程执行`run()`

<h3><code>start</code></h3>

```
void start()
```

- 启动线程，使线程从`New`状态进入`RUNNABLE`状态。
- `start()`只是让线程进入`Java`层面的`RUNNABLE`状态,是否运行线程由操作系统的调度器决定
- 每个线程对象的`start()`只能调用一次，如果调用了多次会出`IllegalThreadStateException`

<h4><code>join</code></h4>

```java
//无限等待,直到线程执行完毕
void join();
//限制最大等待时间为millis毫秒，如果超过等待时间则继续向下运行
synchronized void join(long millis)
```

- 让当前线程等待目标线程执行完成后再继续执行。
- 底层基于`Object.wait`实现，会使当前线程进入`WAITING`或`TIMED_WAITING`状态。

<h4><code>interrupt</code></h4>

```java
void interrupt()
```

- 向目标线程发送一个中断请求，本质是设置中断标志位。
- 如果线程处于阻塞状态，会唤醒阻塞线程，清除中断标志位，抛出`InterruptedException`异常。
- 如果线程处于可运行状态，则仅仅设置中断标志位,可通过轮询中断标志位来决定是否要停止线程。

<h4><code>isInterrupted</code></h4>

```java
boolean isInterrupted()
```

- 判断目标线程是否被中断，不会清除中断标志。

<h4><code>setDaemon</code></h4>

```java
void setDaemon(boolean on)
```

- 将线程标记为守护线程，用于提供后台服务。
- 守护线程与用户线程的区别是:用户线程全部结束后，即使守护线程未执行完毕，JVM也会强制退出
- 垃圾回收器线程就是一个守护线程

## 创建线程的方式

**继承Thread类并重写`run()`方法，调用`start`启动线程**

```java
        //创建线程对象
        Thread myThread = new Thread("t1") {
            @Override
            public void run() {
                System.out.println("执行线程");
            }
        };
        //启动线程
        myThread.start();
```

**优点**

- 编写简单，在方法中可以直接通过`this`获得当前线程，无需使用`Thread.currentThread()`。

**缺点**

- 线程类已经继承了`Thread`类，无法再直接继承其他类
- 线程要执行的任务写在`run`方法中无法更改，不利于线程与任务解耦

------

**实现Runnable接口，将`Runnable`对象作为参数传入`Thread`类构造器，创建实例后调用`start`方法启动线程**

```java
   Runnable runnable = new Runnable() {
            @Override
            public void run() {
                log.info("hello");
            }
        };
        Thread thread = new Thread(runnable,"t1");
        thread.start();
```

- 本质仍是通过`Thread`类的`run()`方法执行
- `Runnalbe`是一个任务类,更易于与线程池等高级API配合

**实现`Callable`接口，通过`Callable`对象初始化`FutureTask`对象,将`FutreTask`对象作为参数传入`Thread`的构造器，创建实例后调用`start`方法启动线程**

```java
        FutureTask<Integer> task = new FutureTask<>(new Callable<Integer>() {
            @Override
            public Integer call() throws Exception {
                log.info("hello,World");
                return 100;
            }
        });
        Thread thread = new Thread(task,"t1");
        thread.start();
        Integer result = task.get();
```

- `FutureTask`也是一个任务类,继承自`Runnable`接口
- `FutureTask`的优点是可以在任务执行完毕后,通过`get()`方法获得执行结果
- `call`方法与`Runnable`接口中的`run`方法一样，都是线程执行的任务，但是`call`方法有返回值并且可以抛出异常。

## `wait/notify/notifyAll`

由`Object`提供给Java中所有对象的一组线程通信方法，底层基于对象关联的 `Monitor` 实现。用于配合 `synchronized` 实现多个线程之间的等待与唤醒机制。

- `wait`、`notify`、`notifyAll` 这三个方法只能在当前线程持有该对象关联的` Monitor `时调用，因此必须使用在以该对象为锁对象的 `synchronized` 同步代码块或同步方法中。

<h4>原理</h4>

`Monitor`中有一个等待队列`WaitSet`

**`wait`**

当线程调用`wait`方法时:

1. 检查是否持有 `obj` 的 Monitor,否则抛出 `IllegalMonitorStateException`
2. 释放 `obj` 的 Monitor（锁）
3. 将当前线程加入 `obj` 的等待队列（WaitSet）
4. 线程进入阻塞状态

**`notify`**

当线程调用`notify`方法时：

1. 检查是否持有 `obj` 的 Monitor,否则抛出 `IllegalMonitorStateException`
2. 从 `obj` 的 **WaitSet** 中选择一个等待线程
3. 将该线程从 WaitSet → EntryList（锁竞争队列）

- `notify` **不会释放锁**
- 被唤醒的线程**不会立刻执行**，只有当前线程退出同步块、释放 Monitor 后，被唤醒线程才能重新竞争锁并继续执行

<h4>方法说明</h4>

**`wait`**

```java
//会无限等待,使线程进入WAITING状态
wait()
    
//有时限的等待，使线程进入TIMEWAITING状态，如果在时间内没有其他线程唤醒，则自动唤醒
//如果timeout值为0，也会使线程进入无限等待
wait(long timeout)
```

- 使当前线程释放 `Monitor` 并进入等待队列被阻塞

**`notify`**

```
notify()
```

- 从等待队列中唤醒一个线程，使其重新参与 Monitor 的锁竞争

**`notifyall`**

```
notifyall()
```

- 唤醒等待队列中的所有线程,使其重新参与该 Monitor 的锁竞争。

<h4>正确使用<code>wait/notify</code></h4>

**使用模板**

```java
synchronized(lock）{
	while(条件不成立）{
		lock.wait();
	}
	//干活
}

//另一个线程
synchronized(lock){
    //修改条件
	lock.notifyAll();
}
```

- 等待线程通过`while`循环判断条件是否成立，防止被错误唤醒。被错误唤醒后会重新判断条件是否成立，不成立则继续等待。

**`wait`与`sleep`的区别**

- sleep是Thread方法，而wait是Object 的方法
- sleep不需要强制和`synchronized`配合使用，但wait需要和`synchronized`一起用
- sleep在睡眠的同时，不会释放对象锁，但wait在等待的时候会释放对象锁。

# 线程安全类

> **当多个线程同时访问同一个类的同一个实例时，不会出现数据错误、状态不一致或异常行为**，这种类称为线程安全类

- 线程安全问题的本质原因就是多个线程对共享内存进行非受控的写操作，因此保证线程安全的目标就是防止多个线程同时对共享数据进行非受控的写操作。

**java中，通常使用两种方式实现线程安全类:①使用同步机制（如 `synchronized`）②设计为不可变类**

| 实现方式                      | 原理         | 是否允许写操作 | 性能 | 典型代表                                              |
| ----------------------------- | ------------ | -------------- | ---- | ----------------------------------------------------- |
| **同步机制 (`synchronized`)** | 限制访问顺序 | ✅ 有，但受控制 | 一般 | `Vector`, `Hashtable`, `Collections.synchronizedList` |
| **不可变类 (Immutable)**      | 禁止修改状态 | ❌ 无           | 高   | `String`, `Integer`, `LocalDateTime`                  |

# JMM

JMM(Java内存模型)是一套解决多线程并发问题的规范，规定了多线程环境下线程访问共享变量的行为。核心是规范多线程环境下线程访问共享变量时的原子性，可见性，有序性

- JMM 只是一套规范，不提供具体实现，是由JVM根据规范的要求实现了 `volatile`、`synchronized`、原子类等机制，保证多线程环境下线程访问共享变量时的原子性、可见性和有序性。

<h4>原理</h4>

JMM定义存储共享变量被多线程共享的内存为主内存；每个线程自己独立掌管的内存为工作内存。线程在操作共享变量前，必须从主存中加载到工作内存，操作完成后再写回主内存。然后通过`volatile`、`synchronized`等关键字，控制加载、写回的时机，以及禁止不合理的指令重排，最终保证多线程操作共享变量时能正确交互。

<h4>场景</h4>

线程在操作变量时，会先把主存里的变量读到自己的工作内存中，如果线程执行的是修改操作，修改后的变量可能没及时从工作内存写到主存，此时其他线程读到的就是主存中旧的值。JMM规定了`volatile`关键字保证被修改的共享变量会立刻写回主存，同时让其他线程的缓存失效，必须重新从主内存读，这就保证了可见性

线程在修改共享变量时，会经过三步操作，读共享变量，修改共享变量，将修改后的值写回共享变量。如果有多个线程对同一个共享变量进行操作，这三步在不同线程中交错执行，会导致结果不可预见。JMM规定了`synchronized`保证原子性。加了锁之后，同一时间只有一个线程可以修改共享变量，中间不会被别的线程影响。

编译器或者CPU为了提高执行速度，会在不影响单线程运行结果的情况下调整指令顺序。如初始化A对象，再将A的引用赋值给a，可能被重排为先赋值引用再初始化。这在单线程下是没有问题的，但是在多线程情况下，可能出现另一个线程拿到引用a时，A对象还未被初始化，使得调用出现不可预知的问题。这时可以通过`volatile`或`synchronized`关键字，保证指令的有序性。

## `synchronized`

`synchronized` 是 **JVM 实现的原生互斥锁**，通过对象监视器（Monitor）实现线程互斥。在 Java 代码中通过 `synchronized` 关键字使用。

**特点：**

- JVM 层实现，依赖对象头（Mark Word）+ Monitor（监视器）
- 支持 **偏向锁、轻量级锁、重量级锁**，由 JVM 动态优化
- 保证 **原子性、可见性，有序性**
- **不可中断**，可重入，不可设置超时，公平性不可控制
- 条件等待可通过 `wait()` / `notify()` / `notifyAll()` 实现


### 原理

`synchronized`编译之后对应`monitor enter`和`monitor exit`两个指令，这两个指令保证同一时间代码块只能被获得监视器的线程访问，保证了原子性。

- `monitor enter`用于加锁，加锁时会使用读屏障强行从主存重新读取数据，保证线程读取的数据是最新的；
- `monitor exit`用于解锁，解锁时通过写屏障强制将`CPU`缓存中的数据刷新到主存中，保证线程修改后的数据对其他线程立刻可见，保证了可见性

同时，内存屏障防止了指令重排，保证了有序性。

- 内存屏障只保证了临界区内外的指令不会发生重排序，并不禁止临界区内部的指令重排序

### 锁升级

 `synchronized` 基于 **Monitor** 实现，涉及线程阻塞与唤醒和操作系统互斥量，会产生大量不必要的用户态和内核态的切换和系统调用，性能开销较大。

为提升性能，Java 在 **1.6 引入锁升级机制**，JVM 不会一开始就使用重量级的 Monitor 锁，而是根据**实际竞争情况**逐步升级锁状态：**无锁 → 偏向锁 → 轻量级锁 → 重量级锁**。在无竞争或低竞争场景下尽量避免线程阻塞，仅在竞争激烈时才退化为重量级锁，从而在**线程安全与性能之间取得平衡**。

<h4>锁升级过程简述</h4>

最开始对象是无锁状态，当第一个线程访问同步代码块时，JVM将对象头的`Mark word`中的锁标志位设置为偏向锁，并将线程ID记录到`Mark word`中，说明锁偏向于这个线程，线程进入同步代码块只需要检查`Mark word`中的ID即可，无需其他同步操作。

当其他线程来竞争锁时，锁升级为轻量级锁，会采用`cas`加自旋重试的方式不断重新尝试获得锁

长时间的自旋会浪费CPU资源，当线程自旋到一定次数仍未获得锁或者有更多的线程来抢锁，锁就会升级为重量级锁。

#### 无锁

如果JVM没有开启偏向锁功能，则默认对象为无锁状态，此时对象的`Mark Word`格式为

```
|hashcode (25 bits) | age (4 bits) | biased_flag=0 | lock=01 |
```

在JDK1.6之后偏向锁的默认开启的，但是有一个偏向延迟，需要在JVM启动之后的多少秒之后才能开启，这个可以通过JVM参数进行设置，同时是否开启偏向锁也可以通过JVM参数置。

#### 偏向锁

在JDK1.6后，默认启用偏向锁。在对象创建时，会将对象设置为可偏向状态，此时对象的`Mark Word`格式为

```
| epoch (25 bits) | unused (31 bits) | age (4 bits) | biased_flag=1 | lock=01 |
```

这说明偏向锁已启用，但尚未偏向任何线程，又称为匿名偏向。JVM有偏向延迟，即偏向锁不会在程序启动时立刻生效，可以通过JVM参数`-XX:BiasedLockingStartupDelay=0`设置偏向延迟。在偏向延迟前，创建的对象仍然是无锁状态。

当第一个线程进入`synchronized`时，会在`Mark Word`中写入此线程ID，格式为

```
| epoch (25 bits) | thread_id (54 bits) | age (4 bits) | biased_flag=1 | lock=01 |
```

说明锁偏向`thread_id`这个线程。这个`thread_id`会一直存储在对象头中，直到有第二个线程访问`synchronized`。

**加锁**

1. 检查对象是否为可偏向状态，

   - 如果不是可偏向状态，则直接执行轻量级锁的加锁流程；
   - 如果是可偏向状态，检查偏向锁是否处于匿名偏向。

2. 检查对象是否处于匿名偏向状态，

   - 如果是匿名偏向，直接在对象头的`Mark Word`中写入当前线程的ID，加锁成功。

   - 如果不处于匿名偏向状态，则表名偏向锁已经偏向了一个线程，比较当前线程与偏向锁中线程ID是否相等，如果相等则直接进入`synchronized`同步块中，否则进入偏向锁撤销流程。

**锁撤销**

1. JVM 判断是否需要检查偏向线程的栈状态
   - 若无需检查（如偏向线程已结束），可直接撤销偏向，不触发 STW
   - 若需要确认偏向线程是否仍持有锁，则进入 Safepoint，触发 STW
2. 在 STW 下，JVM 检查偏向线程的栈帧
   - 若仍持有锁，则在其栈中补建 Lock Record，并将对象升级为轻量级锁
   - 若未持有锁，则恢复无锁状态，并根据 epoch 决定是否继续支持偏向
3. 竞争线程重新尝试加锁（轻量级锁或重新偏向）

**从JDK17开始，偏向锁被完全取消，锁升级流程为无锁 ->轻量级锁 ->重量级锁**

#### 轻量级锁

当`synchronized`被多个线程访问，且多个线程间竞争锁时,此时会为对象加轻量级锁。对象头的格式为

```
| ptr to LockRecord / stack (62 bits) | lock=00 |
```

**加锁**

1. 线程在栈帧中创建一个`Lock Record`对象，其中包含了一个`Displaced Mark Word`字段，用于保存对象原始的`Mark Word`。
2. 线程使用`CAS`操作尝试把对象头的`Mark Word`替换为`Lock Record`的地址，并在`Displaced Mark Word`中保存对象原始的`Mark Word`。
   - 如果交换成功，则获取轻量级锁成功，此时`Mark Word`指向`Lock Record`
   - 如果交换失败，说明其他线程拥有锁，线程进行自旋(次数不固定，由JVM自行优化)不断尝试获得锁。若自旋后仍失败，JVM 为该对象 **分配 Monitor 对象**，将`Monitor`的Owner置为持有轻量级锁的线程，并把 Mark Word 替换为 Monitor 地址。当前线程进入 **Monitor 阻塞队列**，等待锁释放。

**释放锁**

1. 检查对象的`Mark Word`是否仍然指向当前线程的`Lock Record`
   - 如果不是，则执行重量级锁解锁流程
   - 如果是，线程通过CAS操作将对象`Mark Word`恢复为`Lock Record`中保存的原始`Mark Word`，解锁成功

#### 重量级锁

当多个线程竞争获取锁时，轻量级锁会升级为重量级锁。此时对象头的格式为

```
| ptr to Monitor (62 bits) | lock=10 |
```

**Monitor**

> 监视器(也称为管程),Monitor是一个对象，由JVM提供，是Java中`synchronized`锁实现的关键。

`Monitor`主要有三个字段

- `WaitSet`：一个链表结构，存储`waiting`状态下的线程
- `EntryList`:一个链表结构，存储等待获取锁的线程
- `Owner`：标记当前获得锁的线程

当进入synchronized代码块中，JVM会将与`synchronized`绑定的对象的对象头设置为`Monitor`对象的地址

- 刚开始Monitor中Owner为null
- 当某个线程开始执行同步代码块时，Monitor的Owner会被置为此线程，Monitor中只能有一个
  Owner
- 当Monitor具有Owner时,其他线程来执行代码块就会进入
  EntryList BLOCKED
- 当线程执行完同步代码块的内容，然后唤醒EntryList中等待的线程来竞争锁，竞争的时是非公平的
- 图中WaitSet中的Thread-0，Thread-1是之前获得过锁，但条件不满足进入WAITING状态的线程，后面

## `volatile`

`volatile`关键字用于修饰成员变量或静态成员变量，它有两个作用：一是保证变量对多线程的可见性，二是禁止指令重排序优化。

<h3>原理</h3>

**可见性**

当变量被声明为`volatile`时，会保证对这个变量的写操作立即刷新到主存中，对这个变量的读操作直接从主存中读取，这意味着一个线程修改了`volatile`变量的值，其他线程能够立刻看到这个修改，不会受到线程工作内存的影响。从而确保了多线程环境下对该变量访问的可见性。

**禁止指令重排序**

`volatile`通过编译器在生成字节码时，在指令序列中添加**内存屏障指令**来禁止指令重排序。

JMM规定了四种内存屏障指令，由JVM来实现它们，以屏蔽不同操作系统和硬件内存屏障实现的差异。

|   屏障类型   |           指令示例           |                             说明                             |
| :----------: | :--------------------------: | :----------------------------------------------------------: |
|  `LoadLoad`  |   `Load1; LoadLoad; Load2`   |     保证`load1`的读取操作在`load2`及后续读取操作之前执行     |
| `StoreStore` | `Store1; StoreStore; Store2` | 在store2及其后的写操作执行前，保证store1的写操作已刷新到主内存 |
| `LoadStore`  |  `Load1; LoadStore; Store2`  |  在stroe2及其后的写操作执行前，保证load1的读操作已读取结束   |
| `StoreLoad`  |  `Store1; StoreLoad; Load2`  | 保证store1的写操作已刷新到主内存之后，load2及其后的读操作才能执行 |

在`volatile`变量写操作之前，会插入一个`StoreStore`屏障。保证之前的普通写操作完成，并禁止重排到 `volatile` 写之后。

在`volatile`变量写操作之后，会插入一个`StoreLoad`屏障。保证 `volatile` 写对其他线程立即可见，并禁止后续操作重排到前面。

在volatile变量读操作之前，会插入一个`LoadLoad`屏障。保证之前的普通读操作完成，并禁止重排到 `volatile` 读之后。

在volatile变量读操作之后，会插入一个`LoadStore`屏障。保证 `volatile` 读完成后，后续普通写操作不重排到前面。









- 可以保证有序性和可见性，无法保证原子性

- 在`sychronized`中操作共享变量也可以解决可见性问题，但是`volatile`更加轻量化，效率更高
- `voaltile`适合于只有一个线程写共享变量，其余线程读共享变量的情况，因为他不能解决原子性问题
- `synchrinized`既可以保证代码块的原子性，同时也能保证代码块内变量的可见性，缺点是性能较低

# LockSupport

Java中用于线程同步的工具类

### 常用方法

<h4><code>park/unpark</code></h4>

```java
//使当前线程等待，进入WAITING状态
static void park(); 
//恢复线程继续运行
static void unpark(Thread thread)
```

- 在线程调用`park`之前调用`unpark`，线程在`park`后会继续执行，不会暂停

**与`Object`的`wait&notify`相比：**

- `wait&notify`必须配合`synchronized`一起使用，而 `park&unpark`不必
- `unpark`可以精确唤醒一个线程，而notify只能随机唤醒一个等待线程，`notifyAll`只能唤醒所有等待线程
- `park&unpark`可以先`unpark`，此后再对此线程调用`park`不会等待；而`wait&notify`先调用`notify`没有任何意义。

**实现原理**

每个线程都会关联一个`Parker`对象，这个对象由`C++`实现，Java层面无法直接操作。`park&unpark`依赖线程的`Parker`对象实现。

`Parker`由三部分组成:

- `_counter`：许可计数器，表示当前线程是否拥有“许可”。只有0或1两个值
- `_cond`:条件变量，用于在 `_counter == 0` 时，使线程进入等待状态，以及在 `unpark` 时唤醒阻塞线程
- `_mutex`：互斥锁，用于保护 `_counter` 与 `_cond` 的并发访问，避免竞态条件和信号丢失

`_counter`初始为0

调用`unpark`时，若 `_counter == 0`，唤醒与该 `Parker` 关联的线程，并将 `_counter` 置为 1；若 `_counter == 1`，说明许可已存在，不做任何操作

调用`park`时，若 `_counter == 1`，直接消费许可并返回,否则阻塞与`Parker`关联的线程。

# 共享模型

> **共享模型**是指： 在多线程程序中，**多个线程通过共享同一块内存区域来通信和交换数据**。

**共享问题**

> 多个线程访问共享资源时，可能会出现一些意想不到的问题

- 多个线程同时读共享资源，不会出现共享问题
- 多个线程同时对共享资源进行读写操作，由于线程上下文切换可能会导致出现不同线程对共享资源操作的指令交错执行，可能会出现问题

**临界区**

> 一段代码块里如果存在对共享资源的多线程读写操作，则称这段代码块为临界区

**竞态条件**

> 多个线程在执行临界区代码，由于代码的执行序列不同而导致结果无法预测,称之为发生了竞态条件

## 线程安全解决方案

> 解决方案可分为两类:阻塞式与非阻塞式

# 死锁

指多个线程在执行过程中，**互相无限等待对方持有的且永远无法释放的资源而陷入永久阻塞** 的状态。

**死锁产生的四个必要条件**

- **互斥条件（Mutual Exclusion）：**资源在同一时刻只能被一个线程占用。
- **持有并等待条件（Hold and Wait）：**线程在持有至少一个资源的同时，又等待其他被占用的资源。
- **不可剥夺条件（No Preemption）：**线程已获得的资源在自愿释放前，不能被其他线程强行剥夺。
- **循环等待条件（Circular Wait）：**多个线程之间形成一种头尾相接的循环等待资源关系。

死锁产生后，如果没有外力干预，这些线程将无法继续执行下去。


## 保护性暂停

> 是同步模式中的一种，用在一个线程等待另一个线程的执行结果时

![image-20251006133553340](D:\笔记\图片\image-20251006133553340.png)

- 有一个结果需要从一个线程传递到另一个线程，让他们关联同一个 GuardedObject
- 如果有结果不断从一个线程到另一个线程那么可以使用消息队列（见生产者/消费者）
- JDK 中，join 的实现、Future 的实现，采用的就是此模式
- 因为要等待另一方的结果，因此归类到同步模式

```java
class GuardedObject {
private Object response;
private final Object lock = new Object();
public Object get() {
synchronized (lock) {
// 条件不满足则等待
while (response == null) {
try {
lock.wait();
} catch (InterruptedException e) {
e.printStackTrace();
}
}
return response;
}
public void complete(Object response) {
synchronized (lock) {
// 条件满足，通知等待线程
this.response = response;
lock.notifyAll();
}
}
}
```

# 并发控制策略

**并发控制策略**是系统对并发冲突的**预期假设及其处理方式**。 常见的并发控制策略包括**悲观锁、乐观锁**等。

### 悲观锁

> 对并发保持悲观态度，认为一定会发生并发冲突，因此线程在操作共享资源前先对该资源加锁，确保整个操作期间**独占或限制其他线程的访问**

- `Synchronized`,`Lock`都属于悲观锁

**优点**

- 实现简单，完全解决了并发冲突

**缺点**

- 多线程串行执行，性能一般

### 乐观锁

> 对并发保持乐观态度，认为不一定会发生并发冲突，因此无需独占和锁定资源。但更新数据前，会检查是否有其他线程修改了该数据。如果没有修改则认为是安全的，继续更新数据；如果已经被修改则说明发生了并发冲突，会放弃此次更新操作，可以重试或抛出异常。

**优点**

- 性能优异，与不加锁基本一致

**缺点**

- 更新成功率在极端情况下会特别低

#### 实现

|    特性    |        值比较法（Value-Compare）         |               版本号法（Version-Number）               |
| :--------: | :--------------------------------------: | :----------------------------------------------------: |
|  核心逻辑  |      检查业务字段值是否满足更新条件      |                检查 version 是否未变化                 |
|  适用场景  | 业务逻辑本身可以决定是否更新（如库存>0） | 需要严格检测任何修改，保证记录一致性（多字段、多操作） |
|  ABA 问题  |                 可能发生                 |                        可以避免                        |
|  重试次数  |       少（只要业务条件成立就更新）       |              多（version变化就需要重试）               |
| 并发复杂度 |                   简单                   |                  较复杂，涉及版本管理                  |

**由于只要数据被其他线程修改，乐观锁就会认为出现并发冲突而停止更新，导致其更新成功率可能会很低。在一些特殊情况下，可以对乐观锁进行优化**

- 在更新时，通过业务条件判断来限制更新，而不是严格的数据变化。如库存超卖问题，只要更新时库存大于0，就可以更新。

# 锁

> **锁（Lock）** 是一种同步工具，用于控制多线程对共享资源的访问顺序，通过在访问资源前获取锁、访问后释放锁，保证在任意时刻**同一资源只能被允许的线程或进程按照规定的顺序访问，从而避免并发冲突**。

## 分类

**按锁的作用范围分类**

### 本地锁

> 本地锁是只在当前JVM进程内生效的同步机制，用于控制同一进程内多个线程对共享资源的互斥访问。

### 分布式锁

> **分布式锁**是一种在多个进程或多台服务器之间提供全局互斥访问的同步机制，用于确保分布式系统中各节点在访问共享资源时能够保持数据一致性和互斥性。

**分布式锁如何协调多个服务器间的并发问题**

分布式锁提供了一个可被所有服务器访问的全局共享节点，这个节点保存了锁状态，多台服务器按顺序请求共享节点获取锁，从而保证同一时间只有一个服务器访问共享资源，解决跨服务器并发问题。

**从线程获取锁的顺序是否公平进行分类**

### 公平锁

按照线程**请求锁的先后顺序**获取锁，先到先得。

- 避免线程饥饿
- 但上下文切换多，**吞吐量较低**。

## 非公平锁

允许线程**插队**抢占锁，不保证线程按请求顺序获取锁。

- 减少线程切换，**吞吐量更高**，
- 可能出现线程饥饿。

# JMM

> Java Memory Model，Java内存模型，是由Java语言规范定义的多线程内存访问规则。定义了主存，工作内存抽象概念，底层对应着CPU寄存器，缓存，硬件内存，CPU指令优化等。

​	JMM主要为了解决以下几个问题:

- 原子性-保证指令不会受到线程上下文切换的影响
- 可见性-保证指令不会受CPU缓存的影响
- 有序性-保证指令不会受CPU指令并行优化的影响

## 可见性

​	主存存储所有共享变量，当线程启动后，最初时，线程每次访问共享变量都会从主存读取所需的值到工作内存；当某个共享变量被多次读取后，JIT编译器会将此变量缓存到线程工作内存的高速缓存中，减少对主存中共享变量的访问，提高效率。

​	此时如果其他线程修改了共享变量的值，此时线程访问的仍然是自己所缓存的旧值，导致了共享变量修改的可见性问题

## 原子性

> 主要解决因为CPU上下文切换导致的指令交错问题

## 有序性

> JIT会在不影响正确性的前提下，调整语句的执行顺序，这称为指令重排。但是在多线程下，指令重排会影响正确性。

**为什么要指令重排**

通过指令重排，可以实现不同指令的并行执行，从而提高执行效率

**禁止指令重排**

将多线程共享变量添加`volatile`可以禁止指令重排。`volatile`可以保证操作此变量之前的代码不会重排序。

# JUC

> Java并发框架，位于`java.util.concurrent`包下，提供了并发编程的一系列工具。

## AQS

> 全称为`AbstractQueuedSynchronizer`，Java中定义的一个抽象类，是JUC中同步工具的基础框架。

**AQS 通过模板方法模式统一定义了“获取 / 释放同步资格”的通用流程，将线程的排队、阻塞和唤醒等并发控制逻辑统一封装。具体的同步语义由继承 AQS 的同步器通过实现 `tryAcquire` / `tryRelease `等钩子方法来完成。**

**JUC 同步工具通常通过组合一个继承自 AQS 的同步器（Sync），将具体同步操作委托给该同步器，从而复用AQS提供的通用排队、阻塞和唤醒机制。**

### 原理

`AQS` 定义了state 字段，用于表示同步器的状态，同步器子类通过对 `state` 的判断决定当前线程是否满足同步条件，其具体语义由子类自行定义。

为了保证`state`在多线程环境下的可见性并防止指令重排，使用`volatile`关键字修饰此字段。

为了保证在多线程环境下修改`state`操作的原子性,提供了基于CPU的`CAS`指令实现的`compareAndSetState`方法修改`state`。

`AQS`中还基于双向链表实现了一个先进先出的等待队列，当线程修改`state`失败时，可以将线程阻塞，封装为链表节点并放入队列中等待被唤醒。

基于此，`AQS`提供了多个钩子方法，子类同步器可以实现钩子方法定义自己的同步语义，而无需自己重复实现等待队列，状态修改等功能。

**特点**

- 提供了条件变量来实现等待，唤醒机制，支持多个条件变量，类似于`Monitor`中的`WaitSet`，只不过可以定义多个`WaitSet`，将线程放入不同的`WaitSet`中精准唤醒。
- `AQS`有独占和共享模式，共享模式下多个线程可以同时持有共享资源，适用于多线程协作场景。独占模式只允许单线程独占资源，适用于线程互斥的场景。

`AQS`的子类需要实现以下父类方法(默认抛出`UnsupportedOperationException`)

- `tryAcquire`:尝试获取锁，成功则返回`true`
- `tryRelease`：尝试释放锁
- `tryAcquireShared`
- `tryReleaseShared`
- `isHeldExclusively`

**ReentrantLock与Synchronized的区别**

- `ReentrantLock`基于`AQS`实现，不但提供了阻塞获取锁的方法，还提供了尝试获取锁的方法，当获取不到锁时可以执行其他操作。`Synchronized`由JVM实现，如果获取不到锁，只能阻塞等待。
- `ReentrantLock`基于`AQS`的等待队列，可以灵活的实现公平锁与非公平锁，而`Synchronized`只能是非公平锁。
- `AQS`有独占和共享模式，共享模式下多个线程可以同时持有共享资源，适用于多线程协作场景。独占模式只允许单线程独占资源，适用于线程互斥的场景。
- `ReentrantLock`支持条件变量，可以将线程放入不同的条件变量中等待，实现更加精细的唤醒通知；`Synchronized`只有一个`WaitSet`集合存放等待的线程。

## `StampedLock`

> 在JDK8加入的读写锁，进一步优化了读性能。

**乐观读**

`StampedLock`基于戳(`stamp`)实现了乐观读的方法，读取完毕后需要做一次戳校验，如果校验通过，则表示期间没有写操作，数据可以安全使用，如果校验没有通过，则需要重新获得锁，保证数据安全。

**特点**

- 不支持条件变量
- 不可重入

## 并发集合

Java提供了一系列用于多线程环境下的集合，相比于`Vector`,`HashTable`等早期线程安全集合，这些集合使用了更先进的机制在保证线程安全的同时性能更高。

- 并发集合只保证单个方法是原子的，不会被其他线程打断。

JUC下的并发集合保证线程安全的策略大致可分为三种，对应实现类的三个关键词：

- `BlockingXXX`:基于锁实现，并提供阻塞的方法
- `CopyOnWriteXXX`：基于 COW 技术实现，写操作开销大，读操作开销小
- `ConcurrentXXX`:使用CAS优化性能，只保证弱一致性
  - 遍历时弱一致性：使用迭代器遍历时，如果容器发生修改，迭代器仍可继续进行遍历，此时内容是旧的
  - 求`size`弱一致性,`size`的返回值不是100%准确
  - 读取弱一致性

对于非线程安全集合，遍历时如果其他线程对其修改，会使用`fail-fast`机制使遍历立刻失败，抛出`ConcurrentModificationException`异常,不再继续遍历。

### `ConcurrentHashMap`

<h4>原理</h4>

**JDK1.7及以前**

`ConcurrentHashMap`依靠分段锁(Segment锁)实现并发安全，一个`ConcurrentHashMap`中包含一个`Segment`数组，默认数组长度16。一个`Segment`类似于一个小的`HashMap`，每一个`Segment`有自己独立的锁，线程在操作时只需要获得其中一个`Segment`的锁，而不需要对整个`HashMap`加锁，使得不同`Segment`之间的操作可以并发进行，提高了并发性能。

并发度的大小取决于`Segment`数组的大小，而数组初始化后，并发度就固定了，整体性能提升有限。

**JDK1.8后**

取消了`Segment`锁，不再有`Segment`数组，且进一步减小锁粒度，每一个哈希槽一个锁。

添加元素时，首先判断容器是否为空，如果为空，则通过`CAS`操作初始化容器。

如果容器不为空，则根据元素的`hash`值计算位置，判断该位置是否为空
- 如果该位置为空，则通过CAS操作设置该节点
- 如果不为空，则使用`synchronized`锁住该位置的头节点加锁，然后进行添加操作

这相当于对每一个哈希槽加锁，进一步减小了锁的粒度。

##### 构造方法

```
ConcurrentHashMap(int initialCapacity, float loadFactor, int concurrencyLevel)
```

### `List`

#### `CopyOnWriteArrayList`

是ArrayList 保证线程安全的变体，其中所有写操作都通过对底层数组进行全新复制来实现，允许存储null元素。当进行写操作时，使用了Lock锁做同步处理，内部拷贝原数组，并在新数组上进行添加操作，最后将新数组替换掉旧数组；若进行的读操作，则直接返回结果，操作过程中不需要进行同步。

**`CopyOnWriteArrayList`底层使用数组存储元素，每次更新数据时，都会复制一个全新的数组，在新数组中执行更新，然后将新数组引用替换老数组，为保证数组的可见性，对数组字段加了`volatile`关键字修饰**

**写操作**

在修改元素时，首先申请一个`RenntrantLock`锁(新版本直接使用`synchronized`锁)，然后获取到原数组，复制原数组并在新数组上进行元素修改，最后替换原数组的引用，将其指向新数组

- JMM规定，对引用的赋值是原子操作，因此即使读操作没有加锁，读线程始终读取一致的快照，要么看到旧数组，要么看到完整的新数组，绝不可能看到不一致的中间状态。

**读操作**

读操作不加锁，直接读取数组中的元素

### `Queue`

#### `ConcurrentLinkedQueue`

是一个适用于高并发场景下的队列，通过CAS实现了高并发状态下的高性能。通常，`ConcurrentLinkedQueue `的性能要好于`BlockingQueue`。

#### `BlockingQueue`

与`ConcurrentLinkedQueue`的使用场景不同，`BlockingQueue` 的主要功能并不是在于提升高并发时的队列性能，而在于简化多线程间的数据共享。`BlockingQueue`提供一种读写阻塞等待的机制，即如果消费者速度较快，则`BlockingQueue`则可能被清空，此时消费线程再试图从`BlockingQueue`读取数据时就会被阻塞。反之，如果生产线程较快，则`BlockingQueue`可能会被装满，此时，生产线程再试图向`BlockingQueue`队列装入数据时，便会被阻塞等待。

### `Dqeue`

#### `LinkedBlockingDeque`

是一个线程安全的双端队列实现。它的内部使用链表结构，每一个节点都维护了一个前驱节点和一个后驱节点。`LinkedBlockingDeque`没有进行读写锁的分离，因此同一时间只能有一个线程对其进行操作

#### `ConcurrentLinkedDeque`

`ConcurrentLinkedDeque`是一种基于链接节点的无限并发链表。可以安全地并发执行插入、删除和访问操作。当许多线程同时访问一个公共集合时，`ConcurrentLlinkedDeque`是一个合适的选择。

# 线程池

线程池（ThreadPool）是一种多线程使用方式。基于池化的思想，预先创建一定数量的工作线程，将它们保存在线程池中，并通过队列统一管理和调度任务执行，避免每次执行任务都频繁创建/销毁线程。

**优点**

- 通过复用线程避免频繁创建和销毁线程，提高系统的并发性能与资源利用率。
- 通过对线程池中最大线程数量的合理控制，可以取得执行时间与内存消耗的平衡。

<h4>为什么线程的创建和销毁开销大</h4>

Java线程模型是一对一的，一个Java线程对应一个操作系统的内核级线程。

创建Java线程需要通过系统调用切换到内核态创建内核级线程，涉及到内核态和用户态的切换，线程资源分配，栈空间的分配。同样的，销毁线程也需要切换到内核态回收为线程分配的资源，所以Java中线程的创建和销毁开销非常大。

- Java21引入了虚拟线程，基于多对多的线程模型，基于用户态的调度，更加轻量级。

|  TIDYING   |  010  |     -      |        -         | 任务全执行完毕，活动线程为0即将进入终结  |

### 构造方法

```java

```

<h4><code>handler</code>(拒绝策略)</h4>

当线程池中的任务队列满后，对后续任务的处理策略。`ThreadPoolExecutor`中封装了一些拒绝策略的默认实现

|                       |                                              |          |
| :-------------------: | :------------------------------------------: | :------: |
|     `AbortPolicy`     | 让调用者抛出`RejectedExecutionException`异常 | 默认策略 |
|  `CallerRunsPolicy`   |               让调用者运行任务               |          |
|    `DiscardPolicy`    |                 放弃本次任务                 |          |
| `DiscardOldestPolicy` |     放弃队列中最早的任务，本任务取而代之     |          |

<h4><code>corePoolSize</code>(核心线程数)</h4>

核心线程数，即线程池中始终维持的线程数量,默认创建后就不会被销毁

- 默认情况下线程池不会在任务到来前创建所有核心线程，而是采用懒加载，任务到来且没有线程空闲且**核心线程数 < `corePoolSize`** 的情况下才创建核心线程。如果想要预热线程池，可以使用`prestartAllCoreThreads()`方法。

<h4><code>maximumPoolSize</code>(最大线程数)</h4>

核心线程以外的线程为救急线程，即 `maximumPoolSize - corePoolSize` 部分。

- 当任务队列满时，线程池不会立即执行拒绝策略，而是会检查是否达到最大线程数：
  - 若未达到，则创建救急线程执行任务；
  - 若已达到，则执行拒绝策略。

- **救急线程执行完任务后**，不会立即销毁，而是等待一定时间（`keepAliveTime + unit`）再销毁。

<h4><code>keepAliveTime|unit</code>(救急线程的空闲存活时间)</h4>

仅针对救急线程，核心线程默认不会受 `keepAliveTime` 控制（可通过 `allowCoreThreadTimeOut(true)` 改变）。

- 救急线程只有在连续空闲一段时间后才会销毁,如果救急线程在存活时间内又接到新任务去执行，那么存活时间 **会被重置**。

<h4><code>workQueue</code>(阻塞队列)</h4>

阻塞队列用于缓存等待执行的任务

| 队列类型              | 特点                   | 使用场景                 |
| --------------------- | ---------------------- | ------------------------ |
| `ArrayBlockingQueue`  | 有界队列，FIFO         | 控制任务提交速度         |
| `LinkedBlockingQueue` | 可选择有界或无界       | 高吞吐量任务队列         |
| `SynchronousQueue`    | 无缓冲区，直接交给线程 | 适合任务多、线程少的情况 |

如果阻塞队列选择有限队列，当任务超过队列大小时，才会创建救急线程；如果选择无限队列，则不会创建救急线程，会一直使用核心线程执行任务。

<h4><code>threadFactory</code>(线程工厂)</h4>

用于创建线程，用于创建线程，可设置线程名称、优先级、是否为守护线程等。

- `FinalizableDelegatedExecutorService`应用了装饰者模式，只对外暴露了`ExecutorService`接口中的基本方法,无法调用`ThreadPoolExecutor`中特有的方法。

<h4>线程池的执行流程</h4>

当任务提交到线程池时，线程池首先尝试将任务交给空闲核心线程执行；

如果没有空闲核心线程且核心线程数未达到 `corePoolSize`，会创建新的核心线程执行任务。若核心线程满且均未空闲，任务会进入阻塞队列等待；

队列满且活跃线程数未达到 `maximumPoolSize` 时，会创建救急线程执行任务；队列满且线程数已达最大值时，会触发拒绝策略处理任务。

<h4>如何避免线程池中的任务丢失</h4>

可设置`CallerRunsPolicy`拒绝策略，如果阻塞队列满了且没有线程空闲，则会由任务的提交线程执行任务。但是由业务线程执行任务可能导致业务线程被拖慢，无法及时处理业务请求。所以只适合于低并发但不想丢弃任务的场景。

对于高并发场景，可以对线程池的任务持久化。自定义拒绝策略，把被拒绝的任务存储到`MYSQL`或`REDIS`或`MQ`等外部存储中，重写线程池的`afterExecute`方法，当线程池执行完一个任务后，判断如果阻塞队列比较空闲，就读取外部存储中持久化的任务放入阻塞队列中再进行处理。

除了重写`afterExecute`方法外，还可以自定义阻塞队列，当线程从阻塞队列中取任务时，判断阻塞队列是否比较空闲，如果空闲，就从外部存储中读取被持久化的任务放入阻塞队列中执行。

## 不可变类

> 顾名思义，不可变类的属性是不可以改变的。

- 线程安全问题的本质是多个线程同时对同一个变量进行写操作，不可变类无法对属性进行写，因此杜绝了线程安全问题

线程安全是指在多线程并发访问共享内存时，对共享数据的读写均能保持一致性，并且程序行为正确。

## 锁

### 悲观锁

> 认为线程安全问题一定会发生，因此在操作数据之前先获取锁，确保线程串行执行

- 应用
  1. Java中的Synchronized，Lock都属于悲观锁
- 缺点
  1. 线程串行执行，性能差，不适合高并发场景

### 乐观锁

> 认为线程安全问题不一定会发生，因此不加锁，只是在更新数据时去判断有没有其他线程对数据做了修改；如果判断没有修改则认为是安全的，自己才会更新数据；如果判断已经被其他线程修改说明发生了线程安全问题，放弃修改，重试或报出异常

- 缺点

1. 只要数据发生修改，就认为会出现线程安全问题，但在某些情况下，数据修改不会导致线程安全问题，从而使得成功率大大降低。(可以通过修改判断修改的条件等方法来降低或避免)

- 实现

## 线程运行原理

> JVM所管理的内存被分为栈，堆，方法区。其中栈就是为线程使用，当线程启动后，虚拟机会为其单独分配一块栈内存

- 每个线程有自己独立的栈空间，用于存储方法调用产生的**栈帧（Stack Frame）**。栈帧就是一块存储局部变量、操作数栈、方法返回地址等信息的内存。
- 每一次方法调用都会在该线程的栈内存中产生一个栈帧，如果线程栈空间耗尽，则会抛出`java.long.StackOverflowError`错误
- 每个线程只能有一个活动栈帧，对应着当前正在执行的方法
- 方法执行完毕后，栈帧对应的内存会被释放掉

<h4>如何停止一个线程的运行</h4>

- **异常法：**在线程的`run`方法中判断当前对象的`interrupted()`状态，停止线程时调用`interrupt()`方法，如果是中断状态则抛出异常，达到中断线程的效果。
- **`stop`暴力停止**：调用`stop`方法会使线程被暴力停止，方法已弃用，该方法会有不好的后果：强制让线程停止有可能使一些请理性的工作得不到完成。
- **使用`return`停止线程：**在run方法中判断当前线程状态，如果为中断状态则`return`。当需要停止线程时，调用`interrupt`方法。

## 线程上下文切换

> 因为一些原因导致CPU不再执行当前线程，转而执行另一个线程，称为线程上下文切换(Thread Context Switch)。导致线程上下文切换的原因有:

- 线程的CPU时间片用尽
- 垃圾回收
- 有更高优先级的线程需要运行
- 线程中调用了`sleep`,`yield`,`wait`,`lock`,`synchronized`等方法
<h3>协程</h3>
协程是一种用户态轻量级线程，由用户程序调度其挂起与执行，而无需涉及用户态与内核态的切换，从而避免线程切换的系统吊用开销。
**原理**
协程本质是用户态轻量级任务单元，通常在少量线程上复用执行。
