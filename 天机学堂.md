# 异常处理

在`CommonExceptionAdvice`异常处理类中定义统一的异常处理方法，再通过自动配置为每个微服务统一引入`MVCConfig`。

## 网关鉴权

网关中通过过滤器(`AccountAuthFilter`)进行登录校验

1. 首先判断该路径是否需要登录，如果无需登录，则直接放行
2. 如果需要登录，则获取请求头中的`Token`。并对其进行校验并解析Token为`UserId`放入请求头`USER_HEADER`中。

这样就从Token中获得到了用户信息，并且将它传递给了下游微服务。

## 用户信息拦截器

在`tj-auth`模块中，定义了`UserInfoInterceptor`拦截器。

在这个拦截器中，获取到了`USER_HEADER`中存放的此次请求的用户ID，并将其存入到一个`ThreadLocal`中，并会在请求结束时将用户ID信息从`ThreadLocal`中移除。

用户信息拦截器定义在公共模块中，基于SpringBoot提供了自动配置功能，引入模块并进行简单配置即可在微服务中开启用户信息拦截功能。

# 多级缓存

为热点且很少发生变动的数据建立多级缓存

- 数据查询时优先查询本地缓存
- 本地缓存不存在，再查询Redis缓存
- Redis不存在，再去查询数据库。

简单来说就是JVM内存的缓存，比如你建立一个HashMap，把数据库查询的数据存入进去。

本地缓存使用Caffeine，这是一个基于Java8开发的，提供了近乎最佳命中率的高性能的本地缓存库。

在公共模块中编写缓存实现类，并通过自动配置将其暴漏给其他微服务。



# 合并写请求

主要是合并更新视频播放进度的请求。

在Redis中通过`Hash`结构将每一个小节的播放进度缓存起来，每个小节都是一个独立的`HashKEY`。其中缓存数据为：记录id，用于根据id更新数据库。播放进度，moment，用于缓存播放进度。播放状态（是否学完），finished，用于判断是否是第一次学完。

每当前端提交播放记录时，我们可以设置一个延迟任务并保存这次提交的进度。等待20秒后（因为前端每15秒提交一次，20秒就是等待下一次提交），检查Redis中的缓存的进度与任务中的进度是否一致。

- 不一致：说明持续在提交，无需处理
- 一致：说明是最后一次提交，更新学习记录、更新课表最近学习小节和时间到数据库中

这里使用JDK自带的`DelayQueue`实现延时队列。

每次更新播放进度的请求到来时，先在缓存中查询是否已经有播放数据，如果有，则返回，如果没有，则更新到缓存中并返回。之后根据缓存与请求数据判断是否需要特殊更新(比如更新此视频已完成等)。如果不需要，则直接将新的进度写入缓存，并在延时队列中添加延时任务，延时任务在20秒后判断在这期间缓存是否被更新，如果没有被更新则执行数据库更新操作。

# 点赞业务

设计了一个通用的顶赞系统

- 通用：点赞业务在设计的时候不要与业务系统耦合，必须同时支持不同业务的点赞功能
- 独立：点赞功能是独立系统，并且不依赖其它服务。这样才具备可迁移性。
- 并发：一些热点业务点赞会很多，所以点赞功能必须支持高并发
- 安全：要做好并发安全控制，避免重复点赞

点赞系统只保存点赞记录，业务方保存数据的点赞数，在保存点赞记录时向业务方发送MQ消息通知业务方修改点赞数。

点赞记录中包含四个主要字段：

- 点赞目标id，也就是业务数据ID
- 点赞人id
- 点赞对象类型（为了通用性），也就是业务类型
- 点赞时间

通过Redis对点赞业务进行优化。通过`Set`类型记录业务的点赞记录，不再使用Mysql数据库。

同时，使用`ZSet`统计总点赞数，每一个业务类型使用一个key，业务ID作为元素，点赞数作为score。并且通过定时任务通过MQ通知定期将点赞总数移除并持久化到数据库中

**为什么不使用Hash**

如果是从节省内存角度来考虑，Hash结构无疑是最佳的选择；但是考虑到将来我们要从Redis读取点赞数，然后移除（避免重复处理）。为了保证线程安全，查询、移除操作必须具备原子性。而SortedSet则提供了几个移除并获取的功能，天生具备原子性。并且我们每隔一段时间就会将数据从Redis移除，并不会占用太多内存。因此，这里我们计划使用SortedSet结构。

**如果点赞数据非常庞大，达到数百亿，Redis使用内存存储能够满足需求吗**

这个时候我们可以将Redis与数据库结合。

- 先利用Redis来记录点赞状态
- 并且定期的将Redis中的点赞状态持久化到数据库
- 对于历史点赞记录，比如下架的课程、或者超过2年以上的访问量较低的数据都可以从redis移除，只保留在数据库中
- 当某个记录点赞时，优先去Redis查询并判断，如果Redis中不存在，再去查询数据库数据并缓存到Redis

# 排行榜

使用Redis的`Zset`结构保存用户积分，并根据积分对用户进行排序。基于`XXLJob`使用定时任务在每月初对上月排行榜数据进行持久化，并清除Redis数据。

基于SpringBoot自动装配和`Nacos`共享配置提供了`XXLJOb`初始配置，无需在每个项目中手动配置。

通过水平分表，每一个赛季的排行榜作为一个独立的表。通过MP的动态表名插件配合`ThreadLocal`，`ThreadLocal`中保存此次持久化的表名，传入动态表名处理器中。

# 优惠券

优惠券可以设置定时方法策略，即在某一个时间开始放发，实现策略有两种：

- 通过`XXL-Job`定时扫描优惠券表，是否有需要方法的优惠券
- 通过延时任务处理。

## 兑换码

可以指定优惠券为兑换码兑换获得。兑换码通过对一个50位的二进制数每5位进行Base32编码转换而得。基于自增Id保证全局唯一。自增Id作为二进制数的低32位，自增Id之前为4位新鲜值，通过新鲜值判断是使用哪一个数组对自增Id进行按位加权。将自增Id每4位为一组，转为十进制通过加权数组加权求和后，拼接到兑换码的高14位。

- 为了保证自增Id全局唯一，自增Id将放入Redis保存，所有服务均可访问到

由于兑换码采用全局自增Id生成，当用户使用兑换码时，可以通过Redis中BitMap的一个Bit位与自增Id映射来查询兑换码是否已经使用

## 优惠券领取

对于优惠券领取在高并发情况下产生的线程安全问题，解决了

**超卖问题**

采用乐观锁，通过`update coupon set issue_num = issue_num + 1 where id = #{id} and issue_num < total_num`在更新时检查库存是否还有剩余来保证数据安全。

**用户限领优惠券数目超过上限**

采用悲观锁，通过`synchronized`(分布式环境下使用分布式锁)对单个用户上锁，减小了锁的粒度，保证了数据安全的同时不影响并发度。

此时产生了锁边界小于事务边界导致锁释放后事务仍未提交，锁失效了。此时减小事务边界，使事务先提交，锁再释放，解决了此问题。

**由于在领取优惠券前需要对领取条件进行多步校验，这需要多次查询数据库，效率低**

采用将优惠券校验条件缓存在Redis中，并将数据库写操作通过MQ变为异步写。

采用Hash类型缓存优惠券需要校验的字段

- 优惠券发放时间
- 优惠券库存
- 用户限领数量

同时再使用Hash类型缓存每一个用户已经领取的优惠券数量，每一张优惠券建立一个key，其中field为每一个用户已经领取的数量。

采用Zset类型缓存兑换码兑换的优惠券的`SerialNum`范围，这是为了快速查询兑换码所对应的优惠券。

```java
redisTemplate.opsForZSet().add(PromotionConstants.COUPON_RANGE_KEY, coupon.getId().toString(), maxSerialNum);
```

由于将写操作变为异步，不得不在发送MQ消息前对领取资格做校验，而不能在领取时校验，这导致不得不加大锁的粒度。

普通领券和兑换码领券都存在这个问题.猜测可以用Lua脚本保证校验和领取的原子性，从而去掉锁。

## 优惠券使用

基于策略模式定义优惠券相关规则，包括优惠券描述，优惠券是否到达使用金额，优惠券折扣金额的计算。使用工厂模式解耦不同策略的使用。

由于优惠券的不同使用顺序会导致最终优惠金额不同，使用回溯算法计算可用优惠券的全排列，并获得最优金额和优惠券使用情况。为加速优惠券的优惠金额计算，使用了线程池并配合`CompletableFuture`和`CountDownLatch`

# 项目亮点

- 基于`SpringBoot`自动配置，为所有微服务模块统一引入异常处理机制于登陆验证机制。
- 基于心跳机制，播放视频时每隔固定时间向服务器提交最新播放进度，实现了视频续播功能。
- 针对高并发写场景(尤其是播放视频时的心跳请求，每15秒都会更新一次视频的播放进度)，采用添加写缓存来合并写请求优化。
- 负责实现学习服务，基于Redis合并写请求和DelayQueue实现断点续播功能，优化高并发的数据库写业务，使误差控制在15秒内。
- 独立设计并实现问答评论模块，根据用户选择的匿名与否，记录问答或者评论信息，并产生对应的积分，通过RabbitMQ推送给积分系统。
- 独立负责点赞服务的搭建与实现，用户前端发送点赞请求到达点赞服务，点赞服务负责存储点赞记录与点赞数量，并通过定时任务发送MQ消息通知业务方更新点赞数量。
- 负责签到功能实现与优化，基于BitMap数据结构存储签到记录，通过MQ推送签到信息及积分到积分服务。
- 负责积分排行榜功能实现，利用Redis的Zset存储本月实时排行榜数据。通过XXL-Job分片任务持久化历史榜单到Mysql数据库中，Mysql采用的是分库分表，利用MybatisPlus的动态表名来计算表名。
- 负责优惠券管理功能，发放优惠券时利用按位加权求和算法和异步线程生成兑换码，并利用BitMap验证兑换码是否已经兑换。保证了兑换码的可读性、不可重兑、防爆刷和高效性。
- 负责优惠券的领取功能，利用乐观锁和分布式锁Redisson解决并发安全问题，解决事务边界原因导致的锁失效问题，通过aspectj动态代理解决事务失效问题。
- 负责实现通用分布式锁的封装，基于AOP、自定义注解、工厂模式和策略模式实现。
- 负责优惠券的使用，通过查询所有可用优惠券，经过初筛、细筛、排列组合计算出所有可用优惠券组合，并基于CompleteableFuture并行计算每种组合的优惠明细，最后按照规则筛选出最优解。

# Redis

通过单线程执行器串行执行命令；命令只有在到达 `CommandHandler` 时才会被提交到该线程执行；而过期删除、内存淘汰等后台维护逻辑则通过定时任务在同一个线程中周期性执行。

- 基于Netty NIO处理网络请求，自定义RESP解析器，对请求进行解析。
- 基于`MappedByteBuffer`和自定义环形阻塞队列实现了AOF持久化功能
- 实现了String，List，Set，Hash，ZSet五种数据类型，基于策略模式加工厂模式封装了四十余种命令
- 底层使用ConcurrentHashMap存储键值对
- 使用了两种过期淘汰机制：在获取Key时检查key是否过期；当内存达到阈值时，定期抽取Key，判断是否过期，过期则删除。
- 基于Netty实现了不同平台选择平台上最佳Channel实现的机制。
- 通过双重检查锁保证单例模式下的并发安全性

- 自定义哈希表，实现了渐进式哈希，防止哈希表扩容时导致阻塞
- 实现了AOF持久化，在启动时读取AOF文件恢复数据。并实现了AOF重写，防止AOF过于冗余。
- 通过多种设计模式对各模块解耦，如使用策略模式与工厂模式对命令进行封装，使用双重检查锁的单例模式保证单例的完整性。
