我目前正在查看项目中的对话控制器的`service`，也就是`RAGChatServiceImpl`,其中的`streamChat`应该是整个项目的核心方法，我目前理解其流程为：

1. 检查对话ID是否合法，如果是不存在对话ID就创建一个新的对话并分配一个对话ID
2. 通过`RagTraceContext`获得任务Id，任务ID会通过`AOP`在对话任务开启时注入到上下文，并通过`SSE`的`META`事件将`conversationId`和`taskId`下发到前端。它的主要作用是流式任务控制 + 可观测追踪。
3. 然后会创建一个`StreamCallback`，它是流式输出接收器。用于负责把模型 token 推给 SSE、落库、完成/取消处理。
4. 加载历史对话信息，然后将当前的用户消息追加到历史信息中。
5. 将对话历史信息和当前问题进行重写：主要是规范化当前用户消息，并拆分为子问题。
6. 通过意图处理器对用户消息的意图进行分析：系统中构建了一个意图识别树，可以把用户的输入先分类，再决定走哪条处理路径。目前涵盖四大类：知识库精准问答、日常闲聊、情感反馈，以及 MCP 工具调用。
7. 进入引导服务：主要用于歧义澄清，如果存在则返回用户提示，让用户明确意图，而不是进行后续检索。
8. 接下来要判断用户的意图：如果每个子问题的意图都仅命中一个`System`节点，说明不需要进行`MCP`调用或`RAG`检索，直接拼接现有提示词给大模型，返回用户模型输出。
9. 接下来正式进行检索，也就是RAG的部分：调用检索引擎生成检索上下文。如果未检索到相关文档，会直接返回提示信息，而不进行回答。
10. 最后会聚合所有相关信息，包括聚合意图 + 检索上下文 + 历史 + 改写问题，拼成结构化 messages，再流式调用 LLM。

# 多通道检索引擎

采用多通道并行检索，使用`CompletableFutrue`配合线程池在多个通道同时并行检索。

抽象`SearchChannel`屏蔽不同检索途径的差异，所有通道检索完成后聚合所有检索结果，交由后置处理器进行处理。后置过滤器目前貌似只是添加到一个集合中，然后通过`for` 循环遍历依次处理，思考一下这相比于责任链模式哪一个是最优方案，是否需要修改。

# 问题重写与拆分

在基于用户问题进行检索前，首先会对问题进行重写并将其拆分为多个子问题。用户提问可能存在用词模糊，不标准，缺少信息等问题，对问题进行重写以使其能够更好的在向量空间中进行匹配。

在问题重写前，首先会将用户问题进行归一化。归一化后，

如果没有开启问题重写机制，则仅仅将用户问题按照常用的问题分隔符拆分(如`?`,`。`,`;`等字符)，然后作为多子问题返回。

如果开启了问题重写机制，则会拼接相关的提示词，调用大模型进行问题重写与拆分。

在构建大模型提示词时，只会携带最近`1~2`轮的`User`和`Assistant`消息(最多携带四条历史消息)，过滤掉`System`消息，避免`Token`浪费。

大模型返回后，对其返回结果进行处理，转换为`JSON`对象进行字段提取，最终提取出`rewrite`和`subs`两个字段构造出最终的重写结果并返回。

# 意图识别与引导澄清

在数据库中存储意图树，并在Redis中进行缓存。在发送对话请求时，将意图树相关信息填充到系统提示词中发送给大模型，大模型会返回用户的可能意图与该意图的评分。

目前只对单子问题的情况进行判断是否需要引导澄清，对于多子问题，实现起来比较麻烦，因此不进行判断。

# 链路跟踪

基于`Spring AOP`对对话请求链路进行追踪。定义`@RagTraceNode`注解用于节点方法。添加了注解的方法执行时，会将其执行信息持久化到数据库中，并提供后台管理界面，可以快速查看链路的执行信息，从而快速定位异常位置，为线上排障和效果调优提供可观测依据。

# 请求限流

基于`Redis`的`Sub/Pub`与`ZSet`和`Redisson`信号量实现的分布式系统下的流式对话限流器，并在方法执行时，通过`SpringAOP`增强方法，将执行逻辑交给限流器。

请求到达 -> 有许可且 Lua 判定可出队则执行 -> 否则进入`ZSet`等待 -> 通过轮询兜底 + Pub/Sub 加速唤醒 -> 成功后执行业务 -> 结束时归还许可并广播通知下一批请求重试。

# 文档入库ETL

提供了两种文档入库流程，一种是`chunk`模式，会进行固定的入库流程：解析、切分、向量化；另一种是`pipeline`模式，由用户配置入库管道的各个节点，自定义文档入库流程，包括解析，AI增强，分块，向量化等，以实现对文档灵活的处理。

- 在`IngestionEngine`中，是基于节点配置链执行一条管道的各个节点的，这导致管道的执行逻辑分散。一个节点是否需要执行，以及怎么执行，都是由`IngestionEngine`手动判断并在执行时动态传入的。是否可以借鉴责任链模式，在执行前基于节点配置链构建可执行节点链，将条件判断与参数绑定前置到构建阶段；执行期仅传入上下文并驱动链路运行。以保证引擎职责收敛、节点职责更清晰、运行时分支判断减少、可测试性与可维护性提升。

# 模型路由

基于模型优先级与健康状态动态路由推理模型，基于请求状态动态维护模型健康状态，流式阶段引入首包探测与缓冲回放，异常时自动降级切换。

基于断路器模式对模型调用进行监控与熔断，防止级联故障，导致服务崩溃。
