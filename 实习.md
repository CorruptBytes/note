## 知识沉淀

针对经营报表中的数据，建立标准化的分析与知识沉淀机制，将不同维度数据对应的自动分析结果、人工业务判断及经验总结进行统一记录，并与原始数据关联保存，形成可追溯、可复用的数据知识资产。

**知识与原始数据的强关联存储与查询**

- 设计了**知识 - 数据关联存储模型**，通过唯一标识将沉淀的分析结论、人工判断与原始报表数据进行绑定，实现了 “点击数据即可查看所有历史分析” 的可追溯能力。

针对唯一标识怎么设计，我们原本是打算通过多个字段映射出一个唯一标识，考虑到范围查询需求，准备建立倒查表，但最后仍然选择了直接存储数据的多个字段作为唯一标识

## 销售报表

为销售部门提供组织维度以及商品维度的经营分析报表，包含日报、周报、月报三种周期。报表需通过可视化图表、详细统计表格，并借助AI能力生成价值点分析，辅助管理层进行经营决策。

1. ## 模块结构

- **分包逻辑**：按照领域设计原则，设置`销售域`，并在销售域下分为`组织维度`报表以及`商品维度`报表；
- **controller****层**：统计图表、数据表格，同一个请求返回，数据表格一次性返回，在前端做分页逻辑，避免重复计算。提供AI价值点分析接口进行异步调用，解决AI响应速度较慢的问题（可考虑使用websocket实时输出？）
- **service层** ：处理聚合逻辑，转换逻辑，并将图表以及数据表格内容转化为AI 所需的数据格式进行缓存，方便AI价值点分析接口进行调用，避免重复计算。
- **repository****层** ：提供基础统计数据查询，缓存保存以及读取（Redis）

1. ## 使用缓存

- **统计结果缓存** ：缓存图表 和 报表数据的统计结果，一是AI分析接口调用时，避免二次统计；二是相同的查询条件可以直接服用
- **AI价值点分析结果缓存** ：存储AI价值点分析结果，一是提升查询效率，二是避免过度的token消耗。

## 依赖倒置具体实践

在后端三层架构(`Controller`,`Service`,`Repository/DAO`)中，采用依赖倒置原则。`Service`层和`Repository`层共同依赖`Data`接口，`Data`接口定义了数据能力。由`Repository`实现`Data`接口，通过`Spring`中的IOC容器注入到`Service`层使用。

```
Controller → Service
Repository → Data接口
Service → Data接口
```

这有两种实现方式：

- `service`层中定义`Data`接口，由`Repository`依赖`service`层并提供实现

  这样的优点是完全屏蔽了`repository`的变化，`service`完全感知不到`repository`。

- 将`Data`接口抽离到单独的模块`domain`中，所有需要此接口的模块都依赖`domain`。

**优点**

- 高层不依赖低层，解耦业务逻辑与数据访问实现，且支持数据访问实现的自由替换
- 方便单元测试，不需要连接数据库，就可以对业务层进行测试
- 符合开闭原则，当修改持久化实现时，只需新增实现类，无需变更已有代码
- 控制权上移，由业务层定义它需要什么样的数据能力，而不是由 Repository 决定提供什么能力，数据层只是执行者，业务层才是规则制定者。

一个 **开源的权限控制框架**，用于在应用程序中实现各种 **访问控制模型（RBAC / ABAC / ACL 等）**。核心思想是把权限规则从散落的代码中抽离到权限引擎中，代码只负责调用权限引擎。

1. 定义权限模型（model.conf）

   ```
   [request_definition]
   r = sub, obj, act
   
   [policy_definition]
   p = sub, obj, act
   
   [role_definition]
   g = _, _
   
   [policy_effect]
   e = some(where (p.eft == allow))
   
   [matchers]
   m = g(r.sub, p.sub) && r.obj == p.obj && r.act == p.act
   ```

2. 定义权限规则（policy.csv）

   ```
   p, admin, user, read
   p, admin, user, write
   p, admin, user, delete
   
   p, user, order, read
   
   g, alice, admin
   g, bob, user
   ```

   - 支持数据库存储

3. 使用`casbin`鉴权

   ```
   Enforcer enforcer = new Enforcer("model.conf", "policy.csv");
   boolean result = enforcer.enforce("alice", "user", "read");
   ```

## ETL

ETL = Extract（抽取） + Transform（转换） + Load（加载），是一种用于**数据整合与处理的流程模型**。本质就是将脏乱分散的数据，变成干净结构化的可分析数据。

广泛用于：数据仓库，报表系统，BI 分析，数据沉淀系统，大数据平台。

它将数据整合与处理分为三个阶段：

<h3><code>Extract</code>(抽取)</h3>

从不同的数据源中获得数据，数据源可能包括:数据库，日志文件，第三方接口，消息队列，Excel等。

<h3><code>Transform</code>(转换)</h3>

把原始数据处理成可分析的数据。包括

- 数据清洗（去重、去空）
- 格式转换（字符串转时间）
- 维度映射（城市编码 → 城市名称）
- 聚合计算（统计销量、利润率）
- 衍生字段计算（环比、同比）

<h3><code>Load</code>(加载)</h3>

把处理后的数据写入目标系统：

- 数据仓库
- 报表系统
- 数据中台
- 分析库

除了ETL，还有ELT。它与`ETL`的区别就是将数据在目标系统中进行转换，因为现代数据库算力很强，所以可以实现。

## WebSocket

是一种在 TCP 基础上建立的持久化、双向、实时通信协议，专门用于在 **客户端（通常是浏览器）和服务器之间建立持久化、双向的实时通信**。用来弥补 HTTP 协议在实时交互上的不足。

<h4>特点</h4>

- **双向实时通信：**不像 HTTP 是请求-响应模式，WebSocket建立连接后，**客户端和服务器可以随时互相发送消息**。
- **长连接：**连接建立后会一直保持，不需要每次交互都重新建立 TCP 连接，降低了延迟和资源消耗。
- **轻量数据传输：**数据帧开销小，不像 HTTP 每次都需要完整的头部信息，适合实时数据传输。
- **实时性强：**适合聊天应用、股票行情、游戏、物联网设备等需要即时数据更新的场景。

<h4>工作原理</h4>

**握手阶段**

- 浏览器通过 HTTP 协议发起一个特殊的请求（带 `Upgrade: websocket`），告诉服务器希望升级到 WebSocket 协议。
- 服务器同意后返回 101 状态码，表示协议切换成功。
- 之后，连接就变成 WebSocket 连接，保持长连接。

**数据传输阶段**

- 双方可以随时发送消息，消息以 **数据帧（frame）** 形式传输，支持文本或二进制。

**关闭连接**

- 任意一方都可以发送关闭帧，关闭连接。

## `HttpOnly Cookie`

`HTTPOnly Cookie` 是一种特殊类型的 **浏览器 Cookie**，它的作用是 **增加安全性**，防止客户端脚本（如 JavaScript）访问 Cookie，从而降低 **XSS（跨站脚本攻击）** 被利用的风险。

`HttpOnly Cookie`是带有`HttpOnly`属性的Cookie，服务端在设置`Cookie`时可以为其添加这个属性

- 带有`HttpOnly`属性的Cookie只允许浏览器在发送相同域名的`Http`请求时自动携带，**JavaScript 脚本无法通过 `document.cookie` 读取这个 Cookie**。

```
Set-Cookie: sessionId=abc123; HttpOnly; Path=/; Secure; SameSite=Strict
```
