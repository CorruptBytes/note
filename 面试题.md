# Redis

<h3>Redis的大<code>KEY</code>怎么处理</h3>

<h4>查找大<code>KEY</code></h4>

**`--bigkeys`工具**

Redis中提供了 **`redis-cli --bigkeys`** 工具，可以扫描出Redis中每一个类型最大的 key。

```
redis-cli --bigkeys
```

在使用命令时要注意：

- 最好选择在从节点上执行该命令。因为主节点上执行时，会阻塞主节点；如果没有从节点，可在业务低峰扫描，或用 `-i` 控制间隔，避免影响正常业务处理。

该命令有以下不足之处：

- 这个方法只能返回每种类型中最大的那个`bigkey`，无法得到大小排在前N位的`bigkey`；
- 对于集合类型，该工具仅统计集合元素个数，而非具体占用内存。而集合元素多不一定占用内存多，因为每个元素可能很小，总内存开销未必大。

**`scan`命令**

`SCAN`命令可以增量迭代遍历数据库中的 key。使用`SCAN`命令配合脚本遍历并处理所有`KEY`以寻找大`KEY`。

对于String类型，可以直接使用`STRLEN`命令获取字符串的长度，也就是占用的内存空间字节数。

对于集合类型来说，有两种方法可以获得它占用的内存大小：

- 如果能够预先从业务层知道集合元素的平均大小，可以使用命令获得集合中元素个数，然后乘以集合元素的平均大小
- 如果不能提前知道写入集合的元素大小，可以使用`MEMORY USAGE`命令（需要Redis4.0及以上版本），查询一个键值对占用的内存空间。

**`RdbTools`工具**

`RdbTools`是一个第三方开源工具，可以解析`RDB`文件，找到其中的大`key`。

```
#将大于10kb的key输出到一个表格文件。
rdb dump.rdb -Cmemory --bytes 1024e -f redis.csv
```

<h4>处理大<code>KEY</code></h4>

- **拆分大 Key**：将集合类型的 `Key` 拆分为多个小 Key，确保每个 Key 成员数量合理。在 Redis 集群中，有助于数据分片和内存平衡。

- **清理并转移大 Key到其他组件**：将不适合 Redis 存储的数据迁移到其他存储，并在 Redis 中异步删除。

- **监控内存水位**：通过监控系统设置报警阈值，例如内存使用率超过 70% 或 1 小时内存增长率超过 20%。

- **定期清理过期数据**：防止过期数据堆积导致大 Key 产生，例如 HASH 类型增量写入的情况，可用定时任务清理失效数据。

<h4>删除大<code>KEY</code></h4>

大`key`占用内存空间大，如果正常使用`DEL`删除，可能会导致`Redis`主线程阻塞。

对于集合类型的`KEY`，可采取分批次删除：

- **大 Hash**：用 `HSCAN` 每次获取 100 个字段，再 `HDEL` 删除。
- **大 List**：用 `LTRIM` 每次删除少量元素。
- **大 Set**：用 `SSCAN` 每次扫描 100 个元素，再 `SREM` 删除。
- **大 ZSet**：用 `ZREMRANGEBYRANK` 每次删除前 100 个元素。

在Redis4.0版本，提供了`unlink`命令，可以异步删除`key`，从而不阻塞主线程。

除了主动调用 `UNLINK` 异步删除外，还可通过配置自动触发 lazy free：

```
# 默认均关闭
lazyfree-lazy-eviction no
lazyfree-lazy-expire no
lazyfree-lazy-server-del no
slave-lazy-flush no
```

- **lazyfree-lazy-eviction**：内存超过 `maxmemory` 时，是否异步删除 key。
- **lazyfree-lazy-expire**：过期 key 是否异步删除。
- **lazyfree-lazy-server-del**：处理已有 key（如 `RENAME` 覆盖）时，是否异步删除。
- **slave-lazy-flush**：从节点全量同步前 `FLUSHALL` 是否异步删除 key。

<h3>如何保证缓存与数据库的数据一致性</h3>

常见的保证缓存和数据库一致性的策略有三种：

- `Cache-Aside`(旁路缓存策略)：应用程序直接与缓存与数据库交互，并负责缓存的维护。适合读多写少的场景

  读操作时，如果读取的数据命中缓存，则直接返回数据；如果读取的数据没有命中缓存，则从数据库中读取数据，然后将数据写入到缓存，并且返回给用户。

  写操作时，先更新数据库中的数据，再删除或更新缓存中的数据。

- `Read/Write Through`(读穿/写穿策略)：将缓存与数据库封装为一个组件，应用只与缓存交互，组件负责与数据库交互。

  读数据时，先查询缓存中数据是否存在，如果存在则直接返回，如果不存在，则由缓存组件负责从数据库查询数据，并将结果写入到缓存组件，最后缓存组件将数据返回给应用。

  写数据时，先查询要写入的数据在缓存中是否已经存在，如果缓存中数据已经存在，则更新缓存中的数据，并且由缓存组件同步更新到数据库中。如果缓存中数据不存在，直接更新数据库

- `Write Behind Caching`(写回策略)：将缓存与数据库封装为一个组件，在更新数据的时候，只更新缓存，同时将缓存数据设置为脏，然后立马返回，并不会更新数据库。对于数据库的更新，会由组件批量异步更新到数据库。适用于写多的场景，如CPU缓存，操作系统文件系统缓存，InnoDB引擎的`Buffer Pool`。

无论何种策略，想要保证缓存与数据库的强一致性是不可能的，只能尽量保证**最终一致性**。因此，最好为缓存数据设置过期时间，以保证不一致数据不会一直存在。

# 并发编程

<h3>多线程使用场景</h3>

多线程主要用于提升吞吐量，降低响应时间，典型应用场景有：
- IO密集型： 网络请求，文件读写，数据库操作。CPU等待IO时可以切换线程，提高CPU利用率。
  通常设置核心线程数为CPU核数 * （1 + IO等待系数），最大线程数为核心线程数的2 ~4倍(根据具体场景)。
- CPU密集型：如并行计算任务，可以充分利用多核CPU提升计算速度。
  通常设置核心线程数为 CPU核数(或 + 1)，最大线程数与核心线程数基本一致。
- 生产者-消费者模型：典型场景消息队列消息处理，加快消息处理速度，防止消息堆积


## 线程池

<h4>线程池的执行流程</h4>

当任务提交到线程池时，线程池首先检查当前核心线程数是否已满。如果 **核心线程数未达到 `corePoolSize`**，线程池会**直接创建一个核心线程**来执行任务。

当核心线程数已达到 `corePoolSize`，新任务会进入阻塞队列等待执行。

如果队列满且当前线程数未达到 `maximumPoolSize`，线程池会创建救急线程处理新任务；

若线程数达到 `maximumPoolSize` 且队列已满，则触发拒绝策略拒绝任务。

<h4>如何避免线程池中的任务丢失</h4>

可以使用 `CallerRunsPolicy` 拒绝策略，当线程池和队列都满时，由提交任务的线程执行任务。但是业务线程执行任务可能导致业务线程被拖慢，无法及时处理业务请求，所以只适合低并发但不想丢任务的场景。

高并发场景下，可对任务进行持久化：自定义拒绝策略，将被拒绝的任务存入 `MySQL`、`Redis` 或 `MQ`，并在 `afterExecute` 方法或自定义阻塞队列的`take`方法中检测队列是否空闲，如果空闲，将持久化任务重新放入队列，从而避免任务丢失。

<h4>动态参数的线程池怎么设计</h4>

线程池提供了一系列 `setXXX` 方法，可以动态修改核心参数。可以将线程池参数配置在配置中心，线程池监听配置变化，当配置更新时，线程池即可动态调整核心参数，实现灵活的线程池管理。

<h4>线程池中线程抛出异常的处理</h4>

对于普通线程而言，如果抛出的异常没有没有在线程内捕获，线程会直接终止。

在线程池中，即使在任务中没有手动捕获异常，`ThreadPoolExecutor`也会对异常进行`try-catch`，防止异常导致线程池中的线程终止。

此时，如果想要手动对异常进行处理，有以下几种方案：

- **`Callable` 任务抛异常：**异常会封装在 **Future** 中,不会抛出给线程池或线程，可以通过`Future.get()`获取异常。

- **`Runnable`任务抛异常：**重写`afterExecute()`或在`ThreadFactory`创建线程时指定` UncaughtExceptionHandler`

  ```JAVA
  //afterExecute方式
  ThreadPoolExecutor executor = new ThreadPoolExecutor(
      2, 4, 60, TimeUnit.SECONDS, new LinkedBlockingQueue<>()) {
      
      @Override
      protected void afterExecute(Runnable r, Throwable t) {
          super.afterExecute(r, t);
          if (t != null) {
              System.out.println("任务抛异常: " + t);
          }
      }
  };
  //UncaughtExceptionHandler方式
  ThreadFactory factory = r -> {
      Thread t = new Thread(r);
      t.setUncaughtExceptionHandler((thread, ex) -> {
          System.out.println(thread.getName() + " 异常: " + ex);
      });
      return t;
  };
  ```

## `synchronized`

<h4>锁锁的到底是什么</h4>

线程锁的本质是对共享资源访问权的控制。对于 `synchronized`，线程竞争的是某个对象的 `Monitor`。只有获取到该 `Monitor` 的线程，才能进入对应的临界区执行代码，从而实现对共享资源的互斥访问。


# Java

## `HashMap`

<h4>JDK7前和JDK8后的HashMap实现变化</h4>

**哈希冲突解决方式**

在 Java 7 及以前，HashMap 的底层由数组和链表组成。当多个 key 经过 hash 计算落到同一个数组下标且不相等时，这些 Entry 会通过 `next` 指针串成链表，查询时要先根据 key 的 hash 定位到数组下标，再在链表中逐个比较 key，冲突严重时查询复杂度可能退化为 O(n)。

Java 8 对此进行了优化，在数组和链表的基础上引入红黑树。当某个桶的链表长度达到 8 且数组容量至少为 64 时，链表会转化为红黑树，使查询复杂度从 O(n) 降到 O(log n)。如果红黑树中的节点减少到 6 或以下，又会退化回链表，从而降低维护成本。这种链表与红黑树的动态切换保证了 HashMap 在高冲突场景下既高效又节省空间。

**扩容**

在JDK1.7及以前，扩容时会将每个元素重新哈希后插入新数组；

JDK1.8时，对这个过程进行了优化：因为数组长度始终为2的幂，只需通过判断 `(hash & oldCapacity)` 是否为 0，即可确定元素是留在原位置还是迁移到 `原位置 + oldCapacity`，无需重新计算 hash，从而显著提升扩容效率。

**链表插入方式**

JDK1.7中的HashMap使用头插法向链表插入元素，在多线程的环境下，扩容的时候有可能导致环形链表的出现，形成死循环。

JDK1.8使用尾插法插入元素，在扩容时会保持链表元素原本的顺序，不会出现环形链表的问题。

<h4>为什么JDK8中HashMap使用红黑树而不是AVL树</h4>

因为`AVL`树是严格平衡的二叉树，维护成本较高，而红黑树平衡策略相对宽松，是实际工程中的最佳折中实践。

<h4>为什么JDK8不直接取消链表，全部使用红黑树</h4>

红黑树相比于链表需要占用更多的内存，且当节点较少时，链表与红黑树的查询效率相差很小。

<h4>HashMap在多线程环境下可能出现的问题</h4>

JDK1.7中的HashMap使用头插法向链表插入元素，在多线程的环境下，扩容的时候有可能导致环形链表的出现，形成死循环。因此，JDK1.8使用尾插法插入元素，在扩容时会保持链表元素原本的顺序，不会出现环形链表的问题。

多线程同时执行put操作，如果计算出来的索引位置是相同的，可能造成`key`覆盖，从而导致元素的丢失。此问题在JDK1.7和JDK1.8中都存在。

## `Object`

<h4>为什么equals和hashcode要同时改写</h4>

这是为了保证基于哈希的数据结构的数据一致性，如`HashMap`等类，其工作原理是先通过 hashCode 定位桶，再通过 equals 判断对象是否相等，如果只重写 `equals` 而不重写 `hashCode`，会导致相等对象分布在不同桶，从而出现查找失败的问题。

因此Java规定，equals和hashcode要同时改写，且要求：`equals`返回`true`时，`hashcode`必须相同；但`hashcode`值相同时，`equals`并不一定返回`true`。

# `MQ`

<h3>消费者接收消息后，业务失败了怎么办</h3>

消息已被消费者接收，但在消费者执行业务的过程中业务失败，这有多种解决方案：

最基础的方案可以引入重试机制，消费者在业务执行成功后再ACK，如果失败就不ACK或NACK，让消息重新投递。

进一步的，可以引入死信队列(DLQ)，如果消息在多次重试后，仍然失败，可以进入死信队列，由人工处理。

同时还要保证在重试过程中的消息幂等性，防止重复消费，这可以通过唯一ID解决。

在高可靠场景下，还可以使用本地消息表实现最终一致性。

<h4>如何保证消息可靠性或者说如何保证消息不丢失</h4>

消息的可靠性要从三个阶段保证：生产者、MQ、消费者。

- 对于生产者端，可能存在生产者向MQ发送消息时丢失。

  可以使用发送确认机制，MQ收到消息后回复生产者一个`ACK`以确认接收到消息，并在生产者端添加失败重试机制，保证消息发送成功。

- 对于MQ端，MQ服务可能出现异常导致宕机。

  可以开启MQ的持久化机制，将消息持久化到磁盘中，防止宕机丢失。同时建立MQ集群，保证服务高可用。

- 对于消费者端，可能存在MQ发送消息给消费者时丢失或消费者获得消息后业务处理出现异常

  可以使用手动ACK，在业务处理成功后再确认。如果失败，则触发重试机制，消息会重新发送。

MQ 通常保证“至少一次投递（At-Least-Once)，因此还需要保证消息的幂等性，防止重复消费；

同时也可以使用死信队列，防止异常消息多次重试。

# `MYSQL`

<h3>MYSQL解决超卖问题</h3>

超卖问题是由于事务并发修改导致库存变负数。

```
SELECT stock FROM product WHERE id = 1; -- 100

-- 两个线程同时执行
UPDATE product SET stock = stock - 1 WHERE id = 1;
```

解决方案有多种：

- 乐观锁

  ```
  -- 利用条件更新，执行后通过更新行数判断是否扣减成功
  UPDATE product
  SET stock = stock - 1
  WHERE id = 1 AND stock > 0;
  
  -- 版本号法(乐观锁升级)
  UPDATE product
  SET stock = stock - 1, version = version + 1
  WHERE id = 1 AND version = 10;
  ```

- 悲观锁

  ```
  -- 在事务开启后立刻通过当前读为记录加锁后再判断是否更新
  SELECT stock FROM product WHERE id = 1 FOR UPDATE;
  ```

  - 性能差，高并发场景下容易阻塞

- 添加缓存

  使用`Redis`缓存库存数据，在`Redis`中扣减库存，通过异步更新将缓存数据同步到MYSQL保证最终一致性。

<h3>索引使用原则</h3>

MySQL 优化器会基于统计信息评估查询代价，选择最优索引：如果多个单列索引可用，会选择效率最高的；联合索引如果能减少扫描、回表或排序，通常会被优先使用；如果使用索引的成本高于全表扫描，即使索引可用，也可能选择直接全表扫描。

- **最左前缀法则：**最左前缀法则的含义是联合索引中左侧字段的索引生效，其右侧字段的索引才有可能会生效。因此使用联合索引时查询条件必须包含左侧列，且中间不能跳列，否则从缺失列开始右侧的索引均会失效。如果在联合索引的某一列使用范围查询，后续列的索引均会失效。
- **索引列上不能运算：**查询条件中不能对索引字段进行运算或函数操作，否则索引将失效。这引申出当字符串字段与数字比较时，会触发隐式类型转换（将列转为数字），相当于对索引列进行函数操作，从而导致索引失效，此时要手动为数字加引号转换为字符串比较。
- LIKE 查询中，前缀匹配（如 `'abc%'`）可以使用索引，后缀或全模糊（如 `'%abc'`、`'%abc%'`）会导致索引失效。
- 使用 `OR` 连接条件时，只有当两侧条件都可以使用索引时，MySQL 才可能使用索引（如索引合并）；否则优化器可能选择全表扫描。因为优化器进行成本评估时可能评估使用索引比全表扫描更慢，所以不使用索引
- **覆盖索引：**索引中包含了查询所需的所有字段，无需再进行回表查询，能够显著提高查询效率。

<h3>索引设计原则</h3>

索引设计时有两个指标：在使查询更快的前提下占用空间更小。

- 不要过度建立索引，只在必要的列上建立索引：通常是 `WHERE`、`JOIN` 中的列，以及经常用于 `GROUP BY` 或 `ORDER BY` 的列；因为索引会占用额外的磁盘空间，并且执行写操作时需要额外维护索引，降低写操作的性能。

- 在多条件查询场景下，优先考虑建立联合索引，可以提高查询效率；因为联合索引中记录更多的字段值，更容易形成覆盖索引，避免回表查询

- 具有外键的列最好建立索引，否则在删除或更新父表数据时，需要全表扫描子表来检查约束，性能非常低。

- `text`,`image`,`bit`等数据类型的列不建议建立索引

- 记录数较少的表，索引效果差，没必要在此建立索引

- 更新频繁的字段不适合创建索引

- 数据区分度低(记录的重复值较多)的列不适合建立索引(如性别)

- **前缀索引：**在保证选择性的前提下，对长字符串字段可使用前缀索引，只索引字符串部分前缀，从而减少索引大小并提升查询效率。

  ```mysql
  create index idx_xxxx on table_name(column(n)) ;
  ```

  - 前缀长度可以根据索引的选择性来决定，而选择性是指不重复的索引值（基数）和数据表的记录总数的比值，索引选择性越高则查询效率越高， 唯一索引的选择性是1，这是最好的索引选择性，性能也是最好的。
  - 前缀索引在搜索时，如果搜索词长度超出前缀索引长度，则使用索引排除不匹配的行，然后检查其余行是否可能匹配


<h3>如何定位慢<code>SQL</code>并优化</h3>

定位慢SQL有多种工具：

- 在数据库层面，可以开启慢查询日志，记录执行时间超过阈值的SQL
- 在应用层层面，可以通过日志工具或监控工具记录SQL执行时间，统计平均耗时，最大耗时等

优化`SQL`时，首先使用`EXPLAIN`查看`SQL`执行计划，分析执行类型，索引使用情况等，可根据查询计划采取以下策略优化：

1. **优化索引：**对必要字段创建索引，删除冗余索引。
2. **优化查询结构:**如使用覆盖索引；索引是否失效；减少子查询，优先使用JOIN；分页查询时避免OFFSET；连表查询时以小表驱动大表，并且被驱动表的字段要有索引，最好通过冗余字段设计，避免连表查询。
3. **缓存优化：**对于热点数据，可放入redis或Caffeine缓存，减少数据库访问压力，提升访问速度(考虑缓存一致性问题)。
4. **分库分表：**对于海量数据导致的慢SQL，只能分库分表，减少单表压力(垂直分表和水平分表)。

<h3>超大分页查询如何优化</h3>

`LIMIT offset, size` 会扫描 `offset + size` 条数据，再丢弃前 `offset` 条，导致大量无效扫描，随着 `offset `增大性能急剧下降。

<h4>解决方案</h4>

超大分页优化的核心思路是避免使用 offset，而是通过索引定位数据范围，减少无效扫描和回表次数。

- 利用覆盖索引和子查询减少回表查询

  ```
  SELECT *
  FROM user_demo u,
  (
      SELECT id 
      FROM user_demo 
      ORDER BY id 
      LIMIT 1000000, 10
  ) t
  WHERE u.id = t.id;
  ```

- 游标分页

  记录上一页的最终位置，直接使用最终位置过滤数据

  ```
  SELECT * 
  FROM user_demo
  WHERE id > last_id
  LIMIT 10;
  ```

  - 无法跨页

- 分库分表

  按时间或 ID 范围分表，查询时通过路由规则直接定位目标分表，从而减少单表数据量

- 业务层限制分页数，或引入`ES`等适合超大分页的组件。

# Spring

<h3>事务失效场景</h3>

**1. 方法不是 `public` 修饰**

**原因**：Spring AOP 的`JDK`动态代理只能对 `public` 方法生效，非 `public` 方法无法被代理，因此事务不生效。

**解决方案**：将事务方法修改为 `public`。

```java
@Service
public class UserService {

    @Transactional
    public void saveUser() {
        // 事务方法必须是 public
    }
}
```

**2. 内部调用事务方法**

**原因**：内部调用使用的是原始对象 `this`，绕过 Spring 代理对象，事务不生效。

**解决方案**

- **拆分成两个类**

  将原版的`service`拆分为业务类和事务类，在事务类上添加`@Transactional`，由业务类调用事务类完成业务

  ```java
  @Service
  public class CouponTxService {
      @Transactional
      public void doReceive(Long userId, Long couponId) {
  
          int count = couponMapper.countByUser(userId, couponId);
          if (count > 0) {
              throw new RuntimeException("已领取");
          }
  
          couponMapper.insert(userId, couponId);
      }
  }
  @Service
  public class CouponService {
  
      @Autowired
      private CouponTxService couponTxService;
  
      public void receiveCoupon(Long userId, Long couponId) {
  
          synchronized (userId.toString().intern()) {
              couponTxService.doReceive(userId, couponId); // ✅ 外部调用
          }
      }
  }
  ```

- **注入自身代理对象调用。**

  ```java
  @Service
  public class UserService {
  
      @Autowired
      private UserService self;
  
      public void outerMethod() {
          self.saveUser(); // 通过代理对象调用事务方法
      }
      @Transactional
      public void saveUser() {
          // 事务方法
      }
  }
  ```

- **通过 AOP 上下文获得代理对象**

  1. 引入`Aspectj`依赖

     ```xml
     <dependency>
         <groupId>org.aspectj</groupId> 
         <artifactId>aspectjweaver</artifactId>
     </dependency>
     ```

  2. 开启AspectJ的AOP功能，并暴露代理对象

     ```java
     @EnableAspectJAutoProxy(exposeProxy = true)
     @SpringBootApplication
     public class PromotionApplication {
        //...
     }
     ```

  3. 使用代理对象调用事务方法

     ```java
     IUserCouponService serviceProxy = (IUserCouponService) AopContext.currentProxy();
     serviceProxy.sell();
     ```

**3. 异常被捕获**

**原因**：事务依赖异常回滚，如果事务方法内部捕获异常，代理对象无法感知到异常，就无法回滚。

**解决方案**：不要在事务方法内部捕获异常，或者捕获异常后手动标记回滚。

```java
@Transactional
public void saveUser() {
    try {
        // 业务逻辑
    } catch(Exception e) {
        TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();
    }
}
```

**4. 回滚异常类型不匹配**

**原因**：默认情况下，Spring 事务只对 `RuntimeException` 及其子类异常进行回滚

**解决方案**：通过 `rollbackFor` 属性指定异常类型。

```java
@Transactional(rollbackFor = Exception.class)
public void saveUser() throws Exception {
    throw new Exception();
}
```

**5. 事务传播行为错误**

**原因**：事务方法嵌套调用时，传播行为设置不正确，可能导致事务未按预期生效。

**解决方案**：根据业务选择合适的传播属性，如 `REQUIRED`、`REQUIRES_NEW`。

```java
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void newTransactionMethod() {
    // 新事务方法
}
```

**6. 当前类没有被 Spring 管理**

**原因**：类未被 Spring 托管，框架不会创建代理对象，事务自然不生效。

**解决方案**：将类交给 Spring 管理，例如使用 `@Service`、`@Component`，并通过依赖注入调用。

```java
@Service
public class UserService {

    @Transactional
    public void saveUser() {
        // 事务方法
    }
}
```

**7 . 多线程导致事务上下文丢失**

**原因**：事务是绑定在当前线程的`ThreadLocal`

**解决方案**：不要在事务中使用多线程

# AI

<h3>你平常如何使用AI</h3>

我在使用 AI 编程工具时，更倾向于“**基于文档驱动开发**”的方式。

首先，在编码前我会先整理一份完整的需求文档，并将其提供给 AI，作为上下文输入。同时我会让 AI 反向检查需求是否存在歧义或缺失，从而进一步完善文档，确保需求是清晰且可实现的。

在需求明确后，我再进入编码阶段。在编码前，我通常会提供一份示例工程或既有项目结构，让 AI 生成的代码能够符合团队的代码规范和风格。

在开发过程中，如果需求发生变化，我会优先更新文档，再基于最新文档让 AI进行代码调整，从而保证代码与需求始终一致。

在前端开发场景中，我也会结合接口文档或数据库结构，让 AI 先生成简单的页面原型（如 HTML + CSS），在确认效果后，再逐步生成完整的前端代码，以提高开发效率和准确性。

总体来说，我的使用方式是：**以文档为核心，驱动 AI 参与需求澄清、代码生成和迭代优化的全过程。**同时我也会对 AI 生成的代码进行必要的校验和优化，避免完全依赖 AI，保证代码质量。
